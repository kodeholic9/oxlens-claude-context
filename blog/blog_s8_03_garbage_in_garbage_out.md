# 쓰레기를 넣으면 쓰레기가 나온다 — AI-Native 텔레메트리가 탄생한 이유

> AI와 함께 코딩하기 시리즈 #3

서버 로그를 복사해서 AI에게 던졌다. "PLI가 왜 이렇게 많아?"

AI가 분석을 시작한다. 로그에서 PLI 관련 라인을 추출하고, 타임스탬프를 비교하고, 패턴을 찾는다. 그리고 결론을 내린다. "subscriber의 네트워크 상태가 불안정해서 PLI가 반복 발생하고 있습니다. NACK 대신 PLI로 에스컬레이션되는 빈도를 줄이기 위해..."

그럴듯하다. 논리적이다. 그런데 틀렸다.

서버 로그에는 A, B, C 세 명이 참여한 구간만 캡처했는데, 클라이언트 콘솔 로그는 D의 것을 던졌다. 서버 로그는 10:00~10:05 구간인데, 콘솔 로그는 10:04~10:10 구간이었다. AI는 서로 다른 참여자의, 겹치지도 않는 시간대의 로그를 받은 거다.

AI가 할 수 있는 건 빈칸을 추측으로 채우는 것뿐이었고, 그 추측이 환각이 됐다.

이런 일이 SDP 문제, 오디오 늘어짐, 트랙 불일치 — 수없이 많은 디버깅에서 반복됐다. 원인은 매번 같았다. 준 정보가 불완전하거나, 포맷이 제각각이거나, 핵심 컨텍스트가 빠져있었다.

**쓰레기를 넣으면 쓰레기가 나온다.** AI의 문제가 아니라 입력의 문제다.

---

## 로그를 던지는 건 답이 아니다

서버 로그는 사람이 읽으라고 만든 텍스트다. 시간 포맷이 제각각이고, 중요한 정보와 노이즈가 섞여 있고, 인과 관계가 드러나지 않는다. 사람도 읽기 어려운 로그를 AI에게 던져놓고 "분석해줘"라고 하는 건, AI에게 오염된 정보를 주는 거다.

이 깨달음이 프로젝트의 방향을 바꿨다. 더 좋은 프롬프트를 쓰는 게 아니라, **AI가 읽을 수 있는 데이터를 처음부터 설계해야 한다**는 결론에 도달한 거다.

---

## AI가 소비할 수 있는 텔레메트리

그래서 텔레메트리를 새로 설계했다. 핵심 원칙은 세 가지였다.

**원칙 1: 절대값이 아니라 변화량(delta).**

`packetsSent: 45,230`은 AI에게 아무 의미가 없다. 이 숫자가 크면 좋은 건지 나쁜 건지, 속도가 빠른 건지 느린 건지 알 수 없다. `packetsSentDelta: 71`은 "최근 3초 동안 71개를 보냈다"는 사실이다. 옆에 `packetsSentDelta: 0`인 참여자가 있으면, 즉시 비교가 된다. 무언가 막혀있다는 게 보인다.

Publish 15종, Subscribe 18종 지표를 전부 delta 기반으로 수집한다. 3초 주기로 이전 값을 빼서 변화량만 보내는 거다.

**원칙 2: AI가 자동으로 이벤트를 감지.**

사람이 로그를 읽으면서 "여기서 fps가 갑자기 0이 됐네"를 발견하는 대신, 텔레메트리 시스템이 자동으로 감지해서 이벤트로 기록한다. `quality_limit_change`, `pli_burst`, `nack_burst`, `bitrate_drop`, `video_freeze`, `loss_burst`, `fps_zero`, `encoder_qp_spike`, `video_recv_stall` — 15종의 이벤트가 자동으로 감지되고 타임라인에 찍힌다.

AI에게 "이 구간에서 뭐가 일어났어?"라고 묻는 대신, "fps_zero 이벤트가 10:03:12에 발생했고, 같은 시점에 loss_burst가 있었다"라고 알려주는 거다.

**원칙 3: 정상과 이상의 기준을 문서화.**

이 프로젝트는 Conference 모드와 PTT(Push-to-Talk) 모드를 모두 지원한다. 같은 수치가 모드에 따라 정상일 수도, 이상일 수도 있다.

PTT 모드에서 `recv_delta=0`은 정상이다. 비발화자의 RTP를 서버에서 차단하는 게 설계된 동작이니까. Conference 모드에서 `recv_delta=0`은 패킷이 와야 하는데 안 오는 것이니 문제다.

이 판별 기준을 `METRICS_GUIDE_FOR_AI.md`라는 문서에 정리했다. AI 전용 문서다. 사람이 아닌 AI가 읽고 판단하기 위한 레퍼런스. 모드별 정상/이상 테이블, 섹션별 파싱 규칙, 오진 방지 체크리스트가 들어있다.

## Contract 체크 — AI가 확인하는 체크리스트

텔레메트리에 하나 더 추가한 것이 Contract 체크다. "이것이 동작하고 있으면 시스템이 정상이다"라는 계약 조건 12항목이다.

```
rr_generated > 0     — RTCP Terminator가 RR을 생성하고 있는가
rr_consumed > 0      — subscriber RR을 차단하고 있는가
sr_relay > 0         — Conference SR을 릴레이하고 있는가
egress_drop == 0     — 큐 포화로 패킷이 버려지지 않는가
```

이것들이 전부 통과하면 시스템의 미디어 파이프라인은 정상이다. 하나라도 실패하면 어디가 문제인지 즉시 특정된다.

AI에게 로그를 던져서 "뭐가 문제야?"라고 묻는 것과, Contract 체크 결과를 보여주면서 "sr_relay가 0인데 왜 그런지 찾아줘"라고 묻는 것은 완전히 다른 수준의 대화다.

---

## 협업이 만든 관측 범위

처음부터 이 모든 항목을 계획한 건 아니었다. 출발은 단순했다. AI에게 구조화된 숫자를 줘야 환각이 줄어든다는 깨달음.

그런데 AI와 텔레메트리를 함께 설계하는 과정에서 예상하지 못한 일이 일어났다. AI가 데이터의 소비자이면서 동시에 설계의 참여자 역할을 한 거다.

흐름은 이랬다. getStats 기반 지표를 스냅샷으로 줬더니 AI가 "loss가 어디서 발생한 건지 구분이 안 된다"고 피드백했다. 구간 손실 분리를 추가했다. 구간 분리를 줬더니 "언제 발생한 건지 시점을 모른다"고 했다. 자동 이벤트 감지를 추가했다. 이벤트를 줬더니 PTT 모드에서 정상 수치를 이상으로 오진했다. 모드별 판별 가이드를 만들었다. 가이드를 줬더니 "서버 파이프라인 자체가 정상인지 한눈에 모른다"고 했다. Contract 체크를 추가했다.

도메인 전문가가 "무엇을 측정해야 하는지"를 결정하고, AI가 소비자로서 "이것만으로는 부족하다"를 피드백하는 루프. 도메인 경험이 부족한 상태에서 혼자 설계했다면, 브라우저 getStats가 기본 제공하는 항목에서 크게 벗어나기 어려웠을 거다. AI가 끊임없이 "더 필요하다"고 요구했기 때문에 관측 범위가 넓어진 거다.

결과를 업계 주요 SFU와 비교하면 이렇다. LiveKit은 Prometheus 기반 서버 메트릭과 ConnectionQuality 3단계를 제공하고, mediasoup은 Producer/Consumer별 getStats + Observer API를 제공한다. Janus는 Admin API로 JSON 통계를 노출한다. 2026년 3월 기준, 공개 문서와 소스에서 확인한 범위다.

| 관측 영역 | LiveKit | mediasoup | Janus | OxLens |
|---|---|---|---|---|
| 클라이언트 지표 수집 | getStats 기반 | getStats 기반 | getStats 기반 | 33종 delta 기반 (pub 15 + sub 18) |
| 서버 지표 | Prometheus (노드 집계) | Producer/Consumer stats (JSON) | Admin API (JSON) | per-participant 7종 + GlobalMetrics |
| 자동 이벤트 감지 | 없음 | 없음 | 없음 | 15종 자동 감지 + 타임라인 기록 |
| 구간 손실 분리 | 전체 loss만 | 전체 loss만 | 전체 loss만 | A→SFU / SFU→B 분리 |
| 시스템 정상 판정 | ConnectionQuality 3단계 | Producer/Consumer score | 없음 | Contract 체크 12항목 자동 판정 |
| 모드별 해석 가이드 | 없음 | 없음 | 없음 | METRICS_GUIDE_FOR_AI.md |
| AI 분석용 스냅샷 | 없음 | 없음 | 없음 | 텍스트 1장 export |
| PTT 전용 진단 | 해당 없음 | 해당 없음 | 해당 없음 | 8종 |

물론 이 비교에는 전제가 있다. LiveKit과 mediasoup은 OxLens보다 훨씬 성숙한 제품이고, 텔레메트리 외의 영역(스케일링, 코덱 지원, SDK 생태계)에서는 비교 자체가 무의미하다. 다만 **관측 가능성(Observability)이라는 한 축**에서, AI와의 협업이 만들어낸 커버리지가 어느 수준인지를 보여주고 싶었다. 오늘까지의 결과이며, 이 표는 계속 갱신될 것이다.

항목이 많다고 좋은 건 아니다. 핵심은 두 가지다. 첫째, 절대값이 아니라 delta로 수집해서 AI가 즉시 비교할 수 있다는 것. 둘째, 전체 합산이 아니라 구간별로 분리해서 "어디에서 발생했는가"를 특정할 수 있다는 것. 이 두 가지가 AI의 분석 정확도를 결정적으로 바꿨다.

## 스냅샷 — AI에게 주는 사진 한 장

이 모든 것을 묶어서 텍스트 하나로 export하는 기능을 만들었다. 어드민 대시보드에서 버튼 하나를 누르면, 현재 상태의 스냅샷이 클립보드에 복사된다.

스냅샷에는 SDP 상태, 인코더/디코더 정보, publish/subscribe delta 지표, 이벤트 타임라인, SFU 파이프라인 카운터, Contract 체크 결과가 전부 들어있다. 한 장의 텍스트에 시스템의 전체 상태가 담긴다.

이 스냅샷을 AI에게 주면, 로그를 던졌을 때와는 차원이 다른 분석이 나온다. 참여자별 delta를 비교하고, 이벤트 타임라인에서 인과 관계를 추적하고, Contract 체크 실패 항목에서 근본 원인을 좁힌다. METRICS_GUIDE_FOR_AI.md를 함께 읽으면 모드별 오진도 방지된다.

## AI에게 오염된 정보를 주면 AI도 쓸모없어진다

이 시리즈의 제목은 "AI와 함께 코딩하기"다. 1편에서는 AI가 틀릴 때 선례를 찾으라고 했고, 2편에서는 AI에게 기억을 주는 시스템을 다뤘다.

3편의 교훈은 더 근본적이다. **AI에게 무엇을 주느냐가 AI로부터 무엇을 얻느냐를 결정한다.**

프롬프트를 아무리 잘 쓰고, 컨텍스트를 아무리 잘 관리해도, 데이터 자체가 쓰레기면 결과도 쓰레기다. 로그가 포맷이 제각각이고, 시간 구간이 안 맞고, 핵심 정보가 빠져있으면 — AI는 환각을 만들 수밖에 없다.

그래서 텔레메트리를 처음부터 AI가 소비할 수 있는 구조로 설계했다. delta 기반 지표, 자동 이벤트 감지, 모드별 판별 가이드, Contract 체크, 텍스트 스냅샷. 이건 단순히 "프롬프트를 잘 쓰자"와는 다른 이야기다. 데이터 아키텍처의 문제다.

AI 시대의 관측 가능성(Observability)은, AI가 읽고 판단할 수 있도록 설계되어야 한다.
