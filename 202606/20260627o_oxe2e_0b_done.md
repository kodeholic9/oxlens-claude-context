## 2. 구현 (커밋 5962c99)

- **bot**: `ROOM_JOIN` select 파라미터 + `join_listen()`(청취 affiliate, select=false) + `listen_rooms`. ROOM_JOIN send 적재(매핑 원천).
- **orchestrator**: 봇별 `home` + cross-room `listen` affiliate. 방별 PID suffix 격리 유지(`_room()`).
- **loader**: owner 방 = `PUBLISH_TRACKS send room_id`(서버 track json 엔 room_id 없음 — scope_snapshot 외 미포함), listen 방 = `ROOM_JOIN send room_id` 집합.
- **equations `crossroom_isolation`**: 받은 publisher ssrc 의 owner 방 ⊆ listen 방(누수=격리 깨짐). single-room/매핑부재 skip.
- **scenarios/conf_crossroom**: 양방향 격리 — botC(home X, listen X) / botE(home Y, listen Y), botA·botB 발화.

## 4. ★서버 결함 발견 — under 분리 (부장님 지시 20260627)

cross-room 다방 청취 동적 수신(한 봇 listen=[X,Y]가 X·Y 둘 다 받는 정상 경로) 시험 중 **서버 결함 적발**:

- **증상**(dump 실측): botD listen=[X,Y] → 서버가 botB(Y) 트랙을 **TRACKS_UPDATE 통지(server_sub)는 정확히 함**. 그러나 **RTP forward 0패킷**(client_sub 미수신).
- **봇 한계 아님 실증**: 같은 **단일 sub PC**로 home 방(X, botA) 동적 publish 는 정상 수신(0xA0000001). cross-room affiliate(Y)만 forward 0.
- **결론**: 서버가 cross-room affiliate subscriber 에게 동적 publish 통지는 하되 **RTP forward attach 를 누락**하는 것으로 의심(정적 collect_subscribe_tracks 경로 ≠ 동적 notify 경로).
- **처리**: 부장님 지시대로 격리 명제(누수)를 먼저 커밋. 다방 청취 완전성(under)은 등식에서 분리 → `GAP-TOPO-crossroom-dynamic-fanout`(known_defects.py) 명시 등록. **서버 forward attach 수정 후** under 등식 재도입.
