# oxe2e 회귀 강화 — 요구사항 문서 (Requirements)

> **출처**: 두 개의 설계 논의 세션.
> - 세션 1 = "Wire 덤프 기반 결정적 회귀 게이트 설계 논의" (§0~§15 + 부록)
> - 세션 2 = "3층 피라미드 + AI-in-loop 사업 성립 조건" (§16~§30)
>
> **목적**: 위 두 세션의 *요지*를 왜곡 없이·누락 없이 요구사항(REQ) 형태로 응축. 논의 과정/헛다리 기록은
> 원본 두 문서에 보존되어 있으므로 여기선 **확정된 요구·제약·기각·미확인**만 추린다.
>
> **상태**: 설계/요구 정의 단계. "코딩해" 전. 부장님(kodeholic) 최종 결정 권위.
>
> **부장님 작업 원칙(세션 2 종료 시 확정)**:
> - 내용이 풍성할수록 검토 누락이 줄어든다 → **압축하지 않는다.** 이 문서도 요지 압축이되 차원은 보존.
> - 맥북 진입 후 순서: **현행 파악 → 각 층(3층 제외) 책임 구분 → 진지한 설계 → 1층 테스트 공간 확장(Rust 관례).**
> - 한 세션이 길어지면 김대리가 산으로 가므로, 보험 세션 유지 + 기록 우선.

## A. 배경과 문제 (왜 이 요구가 생겼나)

### A-1. 통증 (원문 보존)
> "기능 수정 후 기존 기능이 깨지는지 회귀 시험을 해야 되는데, 현 oxe2e는 판정 기준이 너무 협소해서 다
> 통과되고 있다. 그 이후 시험 진행이 무의미할 정도로 너무 잘 뚫린다. 브라우저 시험이 디테일을 잡는
> 목적이긴 한데, e2e가 기본적인 것 정도는 잡아줘야 하는데 그 사이클이 끊겼다."

- 증거: **검은 화면, 재입장 영상 미노출 버그가 oxe2e를 통과한 채로 터졌다.**
- 결과: **브라우저에서 삽질하는 시간이 길어진다.** 이 고리를 끊는 것이 최상위 목적.

### A-2. 근본 원인 (세션 1 §1 — 진단)
- **positive existence 판정**: "왔다/됐다" 단언뿐이라 어지간한 변경으로 안 깨짐. (track 도착 ≠ 영상 시험, 0613 교훈)
- **자기채점 구조**: 봇이 자기가 받은 걸 자기가 채점 → 봇 코드가 틀린 방향으로 깨져도 단언이 같이 틀림.
- **회귀 게이트의 본질 = failability**: 코드가 깨졌을 때 *반드시* 빨간불이 켜지는 단언이 아니면 게이트가 아니라 통과 의식(rubber stamp).

> ⚠ **현행 실측으로 이 진단이 일부 수정됨 → §F 참조.** A-2 는 세션 1 시점의 추론. 실제 코드엔 음성 단언·
> 카운터 교차가 이미 존재. 진짜 약점은 "단언 부재"가 아니라 "있는 단언이 binary(count==0)라 안 떨어짐".

## B. 핵심 설계 원칙 (REQ 의 토대 — 세션 1·2 공통 확정)

| ID | 원칙 | 내용 |
|---|---|---|
| P-1 | **골든 불변식 (범위 아닌 등식)** | 덤프/관측을 *그대로* 기준 삼지 않는다. 휘발성(SSRC값·seq/ts 시작값·ufrag·암호문) 제거 후 **불변식(등식)** 추출. exact diff 금지(늑대소년). 덤프=원료, 기준=등식. |
| P-2 | **채점자를 피채점자 밖으로** | failability 의 메커니즘. 봇 자기보고가 아니라 외부 사실(wire/서버 카운터)로 채점. |
| P-3 | **봇 = 무가공 속기사** | 봇은 send/recv raw 를 *해석 없이* 그대로 적는다. 파싱·판정 금지. "모르면 왜곡 못 함." 판정은 별도 검증기. "기가 막히게 잘 = 보낸 거 그대로·받은 거 그대로"(정교함 아님, 무가공). |
| P-4 | **갈래 B 채택 / 갈래 A 기각** | A(정상 패턴 통계 학습)=회귀 게이트로 독(회색지대·self-fulfilling). B(내가 보냈으니 받는 게 결정적)=등식. **범위가 아니라 등식.** ts/jitter 직관은 A 의 "범위"가 아니라 B 의 "명세된 변환식"으로 흡수. |
| P-5 | **전이 = 연속성 (단, 두 종류)** | 정적이든 전이든 핵심 단언 = "수신자가 보는 스트림의 seq 연속 + SSRC 불변". 단 세션 2 §17.3 교정: egress SSRC 가 전환 내내 하나로 유지되는 α(화자전환·레이어전환)만 시점 불필요. β(full↔half)는 SSRC 가 바뀌어 전환 시점이 필요(시그널링에 있음). |
| P-6 | **인과 타임라인 (진단 게이트)** | FAIL 만으로는 디버깅 시작점이 없다 → 고리 안 끊김. "레이어전환 +100ms 후 갭" 식 인과 유추가 서야 디버깅 지점 명확 + AI 의 상관→인과 착각 봉쇄. 전이=시그널링=망 트래픽이라 **한 파일(단일 시계)에 다 잡힘** → 메타 트랙 별도 불요, 파이썬/검증기가 발라냄. |
| P-7 | **검증기도 음성으로 시험** | 검증기도 코드다. 오염 픽스처(seq 갭 주입·원본 SSRC 누출·TRACKS_READY 삭제)에서 *반드시* FAIL 뱉는지 실행으로 증명. "음성 픽스처에서 안 떨어지는 단언은 단언이 아니다." |

## C. 3층 검증 피라미드 (세션 2 §21·§25·§26 — 최상위 구조)

### C-1. 구조 (확정)
```
3층 브라우저 │ 제3자 디코더만 답: 보이나/디코딩되나/화질            │ 사람(부장님) 사이클
            │ KITE식 간접검증 + framesDecoded + PSNR/SSIM         │ 고비용·고삽질·느림
2층 oxe2e   │ 통합·다봇·시그널링 순서·결함주입 하 불변식           │ 김과장(AI) 사이클
 (Rust봇)   │ 천장 = "디코딩되나" 못 봄(우리 스택+복호skip)       │ ★AI 자력규명 마지막 층★
1층 유닛    │ 디코더·타이밍 없이 상태→결과 등식이 서는 모든 것     │ 김과장(AI) 사이클
 (cargo)    │ 가장 싸고 가장 failable, 결정적                      │ 즉결
```

### C-2. 경계 기준 (세션 2 의 핵심 발견 — REQ 가 아니라 제약)
- **층 경계 = "잡을 수 있나"가 아니라 "누구의 사이클이냐".** 1·2층=AI 사이클, 3층=사람 사이클.
- **2층 = AI 가 사람 없이 원인까지 규명 가능한 마지막 층.** 3층까지 새면 부장님이 끌려 들어와 삽질 = 끊고 싶은 고리.
- **그물코가 안 겹쳐야 한다.** 한 층만 손봐선 안 막힌다. 유닛=순수 로직 등식 / 2층=통합 구조+결함 / 3층=디코딩+화질 — 셋이 *다른* 실패를 잡는다.

### C-3. ★부장님 최종 교정 (세션 2 종료 — 이게 C-1/C-2 보다 상위 권위)★
> 세션 2 김대리 결론("2층이 3층을 수렴, 3층 도달 0에 수렴, 3층 얇게")을 부장님이 **폐기/교정**:

- **REQ-C3a**: "2층이 3층 모두를 수렴" 목표 **폐기.** 가능하다고 보지 않음.
- **REQ-C3b**: **2층까지 두텁게·정밀·확실히 쌓고, 한계점은 *모두* 3층으로 수렴.** (경계에서 줄다리기 안 함 — 2층이 확실히 잡는 것까지만, 못 잡는 건 깔끔하게 3층으로.)
- **REQ-C3c**: **업계 트렌드는 3층을 두텁게 쌓지 2층을 두텁게 하지 않는다.** 가능한 한 노력하되 **업계 트렌드를 따른다.** (자체 이론으로 업계 앞지르려다 gold-plating 함정 회피.)
- **REQ-C3d**: 1·2층 두껍게와 3층 두껍게는 **trade-off 아님** (그물코 안 겹침 — 각자 다른 실패를 잡으니 둘 다 두꺼운 게 정답). 세션 2 §22 "3층 얇게"는 폐기.

## D. 사업 성립 조건 (세션 2 §26·§27·§30 — 최상위 목적)

### D-1. 이건 테스트 전략이 아니라 사업 성립 조건이다 (원문 보존)
> "적은 시간, 저비용, 확실한 원인 규명 및 조치 — 이게 되어야 AI IN LOOP. 1인 개발자 with AI pair 가
> 완성된다. 이거 안 되면 사업할 수 없다."

- **REQ-D1**: 검증의 무게중심을 아래로(유닛) 내리고, 2층을 "AI 가 자력 규명 가능한 마지막 층"으로 두텁게 만들어, **사람 사이클(3층) 의존을 줄인다.** (단 REQ-C3b 대로 "0 수렴"이 아니라 "2층이 확실히 잡는 것까지".)
- **REQ-D2**: 3층은 정확도 정점이나 사업축에선 비용 경계. 거기 샜다는 것 자체가 "AI 가 못 막았다"의 신호. **3층 자동화는 가능하나 고비용(구축+규명) — 안전망이지 일터 아님.** 3층이 울면 그 원인을 *가능한 범위에서* 2층 단언으로 끌어내리는 피드백 입구로 쓴다.

### D-2. 꿈의 크기 = 미측정 한 점 (세션 2 §27)
- **REQ-D3 (측정 의무)**: 사업 성립 여부는 **"부장님 버그의 몇 할이 wire 안쪽이고 몇 할이 디코더 안쪽이냐"**(wire:디코더 비율)에 걸려 있다. 이 비율은 **아직 데이터로 없다(전부 추론).** 맥북에서 부장님이 3층/오프라인 디코더로 규명한 버그 목록을 펴서 wire/디코더로 갈라야 사업 사정거리의 실측치가 나온다.
- **REQ-D3 단서 (김대리 딴지 — selection bias)**: 비율 실측 시 "규명된 버그"만 세면 wire 편향(규명된 것만 기억에 남음). **미해결/흐지부지된 증상도 분모에 포함**해야 비율이 안 속는다.
  - 후보 원료(SESSION_INDEX 실측): 0621 검은화면(미해결), 0622 republish 검은화면(video_replay 규명), 0623~0624 돌림노래/NetEQ collapse(수신 NetEQ 지터증폭 규명, SFU 무죄), 0624c PTT 돌림노래 패킷+A/B(video 키프레임 burst→audio send 밀림→수신 NetEQ 264ms 증폭). ← 이 중 wire/디코더 경계가 비율의 원자료.

## E. 요구사항 (REQ — 무엇을 만들/고칠 것인가)

> 세션 1·2 의 설계 의도를 REQ 로 응축. **§F 현행 실측으로 일부는 "이미 있음"으로 판명** → 각 REQ 에 현행
> 상태 태그 부착. [신설]=새로 만듦 / [정밀화]=있는 걸 강화 / [확인필요]=실물 봐야 확정.

### E-1. 1층 유닛 (cargo test) — "디코더 없이 등식이 서는 모든 것"

- **REQ-1.0 [정밀화]** 1층의 정의는 **"rewrite 한정"이 아니라 "디코더·전송 없이 결정적 등식이 서느냐".** (세션 2 §23 부장님 교정 — rewrite 는 그중 한 칸일 뿐. 김대리가 유닛층 전체를 rewrite 로 과축소한 것 정정.)
- **REQ-1.1 [신설]** §23 5영역 유닛 신설:
  - **① 식별/통지**: track_id 생성 등식(`{user}_{대표ssrc}`), vssrc 발급·복사(PublisherStream::new eager → SubscriberStream.vssrc 복사), TRACKS_UPDATE 페이로드 조립(add/remove·per-user·per-track mid), 조회 키(find_stream_by_track_id/by_vssrc, vssrc=0 가드, 역추출 금지).
  - **② 상태 전이**: ParticipantPhase CAS, SubscriberGate pause/resume 진리표(TrackDiscovery→TRACKS_READY·5s·is_allowed), full↔half 캐싱(개인 stream/mid 보존·active:false 포함·복귀 재송출 대상), mid_pool 재활용 순서(release→add·kind별).
  - **③ 자료구조 정합** (★재입장 미노출 버그가 여기 — rewrite 아님): mid_map↔SubscriberStreamIndex↔mid_pool 3중 정합, release_subscribe_track 본문 순서(mid_map→Index→mid_pool), 인덱스 RCU 일관(by_ssrc/by_vssrc/by_mid).
  - **④ scope/라우팅 판정**: sub_rooms 순회 대상 수집, pub_room cascade(leave→pub_room None→floor release), SCOPE batch 순서(sub_add→sub_remove, 폐기 pub_add/remove 무시).
  - **⑤ rewrite** (한 칸): seq 연속(레이어 전환 전후), temporal drop 보정, translate_rtp_ts, rid→vssrc 매핑, retransmission 매핑 유지.
- **REQ-1.2 [정밀화]** **trigger ≠ transform** (세션 2 §24): 유닛은 *타이밍을 시험하는 게 아니라 타이밍을 입력으로 받아 변환을 시험*한다.
  - trigger(언제 통지가 유발되나: PUBLISH_TRACKS 시점이냐 첫 RTP 도착이냐) = 타이밍 = **유닛 밖(2층)**.
  - transform(그 시점 상태 → 페이로드는 반드시 이것) = 순수 함수 = **유닛**.
  - **수렴 단언(핵심)**: 두 경로(early-register vs RTP-first promote) 최종 상태가 같으면 페이로드도 같다. = "어느 경로로 와도 수신측이 보는 건 같다"를 등식으로. ← 부장님이 두 경로 다 잡고 싶은 진짜 이유.
- **REQ-1.3 [확인필요]** **선결조건**: build_tracks_update 가 두 핸들러에 인라인으로 흩어져 있으면 유닛이 못 문다 → "유닛 부적합"이 아니라 **"순수 함수로 추출하라" 신호.** 인라인에 묻힌 칸은 "1층이 싸다"는 전제와 충돌 → **리팩터 선결 비용 포함.** §23 5영역 각 칸에 "이미 분리됨 / 추출 선결" 태그 필요(작업량 추정 안 속으려면 — 김대리 딴지).
- **REQ-1.4 [정밀화]** Rust 관례로 테스트 공간 확장: 파일 내 `#[cfg(test)] mod tests`(유닛, private 접근) + 필요시 `tests/` 디렉토리(통합, public API).

### E-2. 2층 oxe2e (헤드리스 Rust 봇) — "AI 자력 규명 가능한 마지막 층"

- **REQ-2.1 [정밀화]** 판정의 최소 단위를 **binary(count==0) → 등식**으로 격상:
  - seq 갭 == 0 (localhost 무손실 → 갭=논리 버그). 단 재정렬≠손실 — **seq 집합 완전성**으로 본다(도착 순서 단조 아님 — §17.5).
  - 패킷 개수 등식: 수신 == **이번 run 송신 − gated** (골든 대비 아니라 intra-run — 인코더 비결정성 무관. 단 봇은 canned 라 송신도 결정적, §F-3).
  - ts 단조·변환식(α=원본 relay 동일, PTT=translate_rtp_ts).
- **REQ-2.1 단서 [정밀화]** count 등식은 **gated 까지 결정적인 케이스에서만 성립**으로 추가 격리(김대리 딴지 3). 정적 conf(gating 없음)=성립, 화자전환(gating 타이밍)=count 는 약한 단언. 무게는 구조 불변식(seq 연속·SSRC 집합·누출0 — 개수 아니라 구조라 비결정성에 애초에 안 흔들림)에.
- **REQ-2.2 [신설]** 봇 수신 기록을 **ssrc별 누적 카운트 → per-ssrc seq(+ts) 리스트**로 확장. 현행 봇은 ssrc 카운트만 들고 seq/ts 를 버려서(§F-2) seq 연속성·개수 등식을 잴 *원료 자체가 없음*. 무가공 원칙(P-3) 유지하며 속기 내용에 seq/ts 추가.
- **REQ-2.3 [정밀화]** 케이스 매트릭스(세션 1 §7 + 세션 2 §17.3): 정적 5 + 전이 4(α/β 구분) ≈ 9벌. **명세 엔진은 하나**(SSRC 집합 + seq 연속) + **케이스별 데이터** + **예외 하나**(화자전환 ts 점프 = 핸드오프 갭).
  - 정적: ①음성full ②영상full(non-sim) ③영상full simulcast(SSRC 다중) ④음성half(가상SSRC만·원본누출0) ⑤영상half.
  - 전이 α(SSRC 고정·시점 불요): ⑥레이어전환 h↔l ⑦화자전환(half, ts 예외).
  - 전이 β(SSRC 바뀜·시점 필요·시그널링에서 시점): ⑧full→half ⑨half→full.
- **REQ-2.4 [신설]** 인과 타임라인(P-6): 시그널링+미디어 한 파일·단일 시계 → FAIL 시 "전환 +Nms 후 갭" 출력. 멀티봇 전이는 **단일 프로세스 N-peer** 여야 단일 시계 유지(별 프로세스면 cross-process 정렬 부활 — §17.5).
- **REQ-2.5 [정밀화]** 시그널링 opcode 시퀀스 골든: JOIN→PUBLISH_TRACKS→**TRACKS_READY**(존재 단언 — 검은 화면 근본)→… + codec 계약(intent codec 존재 + 서버 등록==클라 선언==수신 PT).
- **REQ-2.6 [신설/차용]** 결함 주입 + 시드 재현(세션 2 §18.4 DST 양념): "localhost 니까 깨끗"을 *가정*만 하지 말고 일부러 재정렬/드롭 주입 → 불변식 버티나 + NACK/RTX 복구 경로를 회귀에 포함. FAIL 시드 재현으로 바이트 동일 재실행. **풀 DST(VOPR)는 과잉**(우리 sfud Tokio 멀티스레드) — "DST 양념 친 와이어 회귀"까지만.
- **REQ-2.7 [정밀화]** 2층 천장 명시: 봇=우리 스택+복호skip → **"디코딩되나" 영원히 못 봄.** 2층 사명 = 디코딩 *선행조건*(키프레임 도착·codec 등록=SDP 선언·TRACKS_READY=PLI 유발·통지 도달·지터 임계)을 wire 로 끌어내려, 3층엔 "선행조건 다 맞았는데 안 그려지는" 진짜 디코더 버그만 남긴다.

### E-3. 3층 브라우저 — 안전망 (세션 2 §28 — 아직 고민 전)

- **REQ-3.1 [보류]** 3층은 **아직 고민 전**(부장님 명시). 업계 트렌드(KITE식 자동화·framesDecoded·PSNR/SSIM 픽셀 골든) 따라 **두텁게** 쌓되(REQ-C3c), 본 요구 정의 범위 밖. 맥북 진입의 "각 층 책임 구분"에서 **3층 제외.**
- **REQ-3.2 [원칙]** 3층 자동화는 가능하나 고비용 — **규명 도구 아니라 최후 안전망.** FAIL=수리 대상 아니라 "2층이 또 놓쳤다" 신호 → 가능 범위에서 2층으로 번역(REQ-D2).

### E-4. 봇/검증기 인프라 (세션 1 §10·§11 + 세션 2 §17.5)

- **REQ-4.1 [정밀화]** 봇 무가공(P-3): send/recv 단일... 정정 → **평문 탭 3곳**(RTP=SRTP 전/후, DC=DTLS 후, WS=평문). 복호 경계 셋. 각 탭에서 무가공 유지.
- **REQ-4.2 [확인필요]** 봇 송수신 훅 = **pion interceptor 패턴**(BindLocalStream/BindRemoteStream — 세션 2 §18.1)을 oxrtc 에 적용. "훅 지점 미확인"의 업계 표준 답 — 새 발명 불요.
- **REQ-4.3 [정밀화]** 레코드 스키마 = (단조 타임스탬프, 방향, raw). send/recv 가 다른 tokio 태스크라 파일 append 순서 ≠ 와이어 순서 → **타임스탬프가 정렬 권위.**
- **REQ-4.4 [신설]** 검증기 음성 픽스처(P-7): §F-1 데인-버그 검산표의 각 항목마다 손상 덤프 만들어 검증기가 반드시 FAIL 뱉는지 실행 증명. `<측정필요>` 빈칸 = 잔존 갈래 A → 빈칸마다 유도 근거 적고, 못 하면 "약한 단언"으로 격리.

## F. 현행 실측 (2026-06-26 맥북 — 세션 1·2 추론의 검증/수정)

> ⚠ **이 섹션이 본 요구 문서에서 가장 중요한 신규 사실.** 두 세션은 *추론*이었고, 실제 oxe2e 코드를 떠보니
> 일부 전제가 어긋난다. 왜곡 없이 적으려면 이 수정을 누락하면 안 된다.
> 출처: `crates/oxe2e/src/{judge/mod.rs, scenario/mod.rs, bot/{mod,media}.rs, lib.rs}` 직접 read.

### F-1. 이미 있는 것 (세션 1 이 "신설 필요"라 본 것 중 존재 — 헛바퀴 방지)
- **음성(negative) 단언 존재**: `judge::evaluate` (c) 누수 — "수신 ssrc ⊆ 약속 ssrc. 약속 안 된 ssrc 1패킷이라도 도착=위반." simulcast l 누수+self-echo+오fan-out 공통 그물. (P-2 음성 단언이 이미 박힘.)
- **gating 음성+양성**: `evaluate_gating` — floor 구간 OnSelf/OnOther/Off 로 갈라 echo 누수·타화자 미도착·idle 누수 검사, guard band(전환 직후/직전 정당 잔류 제외).
- **전이 보존 단언**: `evaluate_caching` — full→half→full 의 TRACK_STATE active:false/true + **track_id 동봉**까지 검증.
- **검증기 음성 픽스처(부분)**: `judge::tests` 에 `caching_fails_when_active_notify_missing_track_id` — track_id 누락 시 FAIL 함을 서버 의존 없이 고정. (P-7 이 이 한 자리엔 이미 적용.)
- **서버 카운터 교차**: `evaluate_{telemetry,rtx_learned,gate_resume,floor_granted}` — 봇 자기보고 아니라 oxcccd REST 되읽기. 채점자 봇 밖(P-2). 특히 `gate_resume` 가 검은 화면 신호 직접 겨냥.

### F-2. 진짜 약점 (세션 추론과 다른 실제 원인 — 설계 출발점을 바꿈)
- **약점 1 — 판정 최소 단위가 `count == 0`**: `evaluate` 이행 = "1패킷이라도 오면 PASS." seq 연속성·개수 등식·ts 단조 **안 봄.** = 부장님 "잘 뚫린다"의 실체. 단언 부재가 아니라 **있는 단언이 binary 라 회귀가 안 떨어짐.** 검은 화면도 "1패킷은 오니까" 이 층에선 PASS.
- **약점 2 — 봇이 seq/ts 를 버림**: `bot::BotResult.received: HashMap<u32,u64>` = ssrc별 누적 카운트만. `media.rs` 수신 task 가 `parse_rtp_mini` 로 까서 ssrc 만 쓰고 seq/ts/marker 버림. → seq 연속성/ts 단조/전이 연속성을 **잴 원료 자체가 봇에 없음.** (REQ-2.2 의 근거.)
- **약점 3 — 1층(유닛) 거의 빔**: oxe2e 전체 `#[cfg(test)]` = judge caching 2개뿐. §23 5영역(rewrite/식별/자료구조 정합/scope) 유닛 0. (REQ-1.1 의 근거. "Rust 관례 테스트 확장"의 빈자리.)

### F-3. 미확인 해소 (세션 2 가 "맥북에서 볼 것"이라 한 것 중 확인됨)
- **봇 송신 결정성 = canned (live encode 아님)** ✅: `RtpSender::next_packet` 고정 ts_inc·seq 단조·고정 페이로드 크기. VP8 라이브 인코딩 비결정성 **없음.** 갈래 B(입력→출력 등식) 토대 코드로 확인 — "N개 쐈으면 N개"의 N 결정적. (세션 2 §17.2 미확인 해소.)
- **시나리오 구조**: `scenario/mod.rs` = TOML(meta/server/participants/timeline + verify_* 플래그). 타임라인 액션 어휘 = join/leave/rejoin/publish/floor/subscribe_layer/mute/wait/republish. **단 실제 .toml 파일 위치·개수 미확인** → §G.
- **봇은 이미 무가공 속기사(부분)**: 수신 task 가 raw decrypt→parse→카운트. 단 속기 내용이 ssrc 뿐(약점 2). P-3 의 "무가공"은 충족, "충분한 원료"는 미충족.

### F-4. 현행이 세션 설계와 정합하는 부분 (유지)
- 봇 = Rust(oxrtc v3), SRTP 복호 키 보유 → RTP raw 접근 가능(브라우저 봇 분기 헛소동은 세션 2 §20-10 에서 이미 철회됨, 현행이 그 철회를 뒷받침).
- 단일 프로세스 N-peer 구조(`run_bot` barrier 동기화) → 단일 시계(REQ-2.4) 토대 있음.
- per-track mid·room_id 필수·codec 명시 등 wire 계약이 봇에 반영됨(0610~0613 정합).

## H. 기각된 접근 (반복 유혹 차단 — 두 세션 누적)

- **갈래 A (정상 패턴 통계 학습)** — 회귀 게이트로 독(회색지대·self-fulfilling). 등식(B)이 정답.
- **exact diff (덤프 그대로 기준)** — 휘발성 때문에 늑대소년. 불변식 추출이 정답.
- **tshark 를 판정 칸에** — tshark 는 파서지 판정기 아님. 판정은 우리 명세. (애초에 봇 내부 덤프라 tshark 자동 파이프라인에서 빠짐.)
- **메타 트랙 별도 생산** — 전환=시그널링=망 트래픽이라 이미 와이어에 있음. 한 파일·단일 시계.
- **외부 캡처(tcpdump) 전제** — 봇이 wire 를 쥐고 있으니 봇 내부 덤프. (단 RTP raw 는 Rust 봇이라 봇 내부로 됨 — 브라우저 분기 철회됨.)
- **봇을 "정교하게"** — 정교할수록 거짓 위험. 무가공이 정직. "기가 막히게 잘=보낸 거/받은 거 그대로."
- **2층이 3층을 수렴 / 3층 도달 0 수렴 / 3층 얇게** — 부장님 폐기(REQ-C3). 한계는 모두 3층으로, 3층도 두껍게(업계 트렌드).
- **풀 DST(VOPR)** — sfud Tokio 멀티스레드라 과잉. "DST 양념" 까지만.
- **1층을 rewrite 로 한정** — 부장님 교정. "디코더 없이 등식 서는 모든 것"(§23 5영역).
- **타이밍을 유닛에 넣기** — trigger 는 유닛 밖. transform 만 유닛(§24).
- **[택1] 남발** — 다차원 논의를 2지선다로 압축 = 사고 강제종료. 합칠 수 있나/더 깎을 수 있나/안 풀리는 한 점 먼저 돌릴 것.
- **장문 메타-반성** — 짧게 받을 자리 짧게.

## I. 한 줄 결론 (두 세션 누적 + 현행 교정)

> 검증의 무게중심을 **유닛으로** 내리고, **2층을 "AI 가 자력 규명 가능한 마지막 층"으로** 두텁게·정밀·확실히 쌓아

> 버려 등식의 원료가 없음 + 1층이 비어 구조 검증을 2층에 떠넘김"** — 이 셋의 정밀화가 첫 실물 작업.
