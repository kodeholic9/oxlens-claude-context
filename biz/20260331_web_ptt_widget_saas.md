## 1. 핵심 발견

### World's First #1: 웹 브라우저에서 동작하는 상용 PTT
- Zello(1.5억 유저), 파워텔, ProPTT2, BizPTT, Voxer, NuovoTeam, VoicePing — 전수 조사 결과 **전부 네이티브 앱 전용**
- Zello Windows PC앱도 2025년 8월 deprecate
- 기술적 이유: PTT는 SFU에서 Floor Control + SSRC rewriting + silence flush + keyframe gating 필요 → 기존 SFU(mediasoup/Janus)에 없음

### World's First #2: 스크립트 한 줄로 웹 서비스에 PTT 무전이 붙는 위젯 SaaS
- 텍스트 채팅 위젯 (Intercom, Crisp, Drift) → 포화
- AI 음성 위젯 (Vapi, Retell) → 생기는 중
- 전화 위젯 (RingCentral Embeddable) → 있음
- **PTT 무전 위젯 → 전 세계 제로**

## 2. PTT vs 통화의 근본적 차이 (제품의 본질)

| 속성 | 통화 | PTT |
|------|------|-----|
| 기본 상태 | 내 마이크 ON (뮤트 깜빡 = 사고) | **내 마이크 OFF (안전)** |
| 항상 연결 비용 | 높음 (소음, 하울링, 뮤트 부담) | **제로** |
| 핵심 경험 | 세션 기반 (걸고, 받고, 끊고) | **항상 청취, 필요할 때 끼어들기** |

→ "옆자리 동료에게 말 거는" 경험의 디지털 버전
→ "Always-on, never awkward."

## 3. 제품 정의

### 통합 방식
```html
<script src="https://oxlens.io/widget.js"
        data-app="abc123"
        data-channel="security-team"
        data-user-token="eyJ...">
</script>
```

### 위젯 형태
- **플로팅 인디케이터** (화면 구석, 최소 크기) — 채널명, 발화자, 연결 상태 표시
- **키보드 단축키로 PTT** — 토글 또는 hold. input/textarea 포커스 시 자동 비활성화
- **마우스 이동 불필요** — 키보드에서 손이 안 떠남. 무전기 PTT 버튼의 본질

### 토글 모드 (AI 대화 등 장시간 발언용)
- 한 번 누르면 발언권 ON → 중간에 얼마든 멈춰도 발언권 유지 → 다시 누르면 OFF
- 구현 난이도 최소, UX 가치 극대

### 기술적 제약
- PC 브라우저: ✅ 모니터 꺼져도 탭 열려있으면 WebRTC 유지
- 모바일 브라우저: ❌ 화면 꺼지면 OS가 WebRTC 연결 끊음
- → **웹 PTT는 "PC 앞에 앉아있는 사람" 전용. 모바일 현장은 네이티브 앱**

## 4. 비즈니스 모델

### 핵심 구조: 위젯은 미끼, SFU + SDK가 본업

```
위젯(무료/저가) → 입소문 → SFU + SDK 판매(B2B) → 디스패치 SI(마지막 카드)
```

- **위젯**: 개발자/웹서비스 업체가 무료로 체험 → 셀프서브, 영업 불필요
- **SFU + SDK**: "더 깊이 통합하고 싶다" → 엔진 라이선스 + Android SDK 판매. 여기서 진짜 매출
- **디스패치 SI**: SFU 고객 중 "관제 화면까지" 요구 시 → 최후의 카드

### 과금
- 최종 사용자로부터 돈을 받는 구조 아님
- **위젯을 도입한 사업자**로부터 B2B 계약
- 초기: 입금 확인 → app_id 발급. 자동 과금 시스템 불필요

### 위젯 SaaS에 필요한 hub 추가 레이어
- 멀티테넌시 (app_id별 고객사 격리)
- 위젯용 토큰 발급 API
- 채널/사용자 관리 CRUD
- hub에 모놀리스로 넣고, 필요 시 분리

## 5. SFU 납품 잠재 고객군

위젯으로 이름이 알려진 후, SFU 엔진을 찾아올 업체들:

| 고객군 | 니즈 | OxLens 매핑 |
|--------|------|-------------|
| MDM 업체 (오피스키퍼 류) | 실시간 화면 스트리밍 (평소 캡처 → 의심 시 스트리밍 ON) | Simulcast (l→h 전환) |
| CCTV/관제 솔루션 | 웹 관제 + PTT 지시 | Conference + PTT + 어드민 |
| 원격 기술지원 | 현장 영상 + 전문가 PTT | 2PC + Floor Control |
| 기존 PTT 업체 | 웹 버전 확장 (자체 SFU 역량 없음) | SFU OEM |

## 6. 디스패치 분해 인사이트

### 안 바뀌는 것 (SaaS 가능 = 엔진)
- PTT Floor Control, 영상/음성 릴레이, 녹화/아카이빙, 위치 추적, 참여자/채널 관리

### 고객마다 바뀌는 것 (SI 영역 = 프레젠테이션)
- 화면 배치, 워크플로우, 외부 시스템 연동, 알림 규칙

### 웹 디스패치의 SI 우위
- WinForm/WPF 기반 디스패치 대비 화면 커스터마이징 공수 1/5
- 배포: URL 하나. 설치/업데이트 없음
- 고객별 패널 조합 가능 (6파일 모듈 구조)

## 7. 검토 후 기각한 아이디어

| 아이디어 | 기각 이유 |
|----------|----------|
| 프록터링(시험감독) | 모니토, 나우앤테스트, 아르고스 등 이미 레드오션. AI 부정행위 감지 기능 필요 → 부장님 영역 아님 |
| 건설/경비/물류 직접 서비스 | 파워텔 독주체제. 전용단말+대리점+현장 엔지니어 네트워크. SKT도 못 잡은 싸움 |
| 웨비나 음성 Q&A 위젯 | 수천~수만 명은 SFU로 불가(CDN/HLS 영역). 큐에 2000명 쌓이면 의미 없음 |
| AI PTT 음성 인터페이스 (독립 사업) | 해자 없음. OpenAI가 버튼 하나 추가하면 끝. 위젯 부가기능으로는 가능 |
| 파트너 채널 (ProPTT2/BizPTT에 SFU 납품) | 자기 솔루션을 못 버림. 시장 파이도 작아서 나눠먹기 어려움 |

## 8. 핵심 강점 정리

- **완성품이 먼저 있고, 아이디어를 나중에 찾는 구조** — 돈을 태우지 않음. 위젯 안 먹히면 피벗, 엔진은 그대로
- **기술 해자** — Floor Control SFU를 바닥부터 Rust로 구현. No fork, no framework. 따라오려면 1~2년
- **1인 All-in-One** — SI 경험(오키토키 프로) + Android SDK + SFU + 웹 클라이언트 + 시그널링. 이 조합 보유자가 업계에 없음
- **영업 팀장 객원 확보** — 전 직장 영업 경력. 제품만 주면 뛰는 구조

## 9. 제품 완전체 로드맵

### PTT만으로는 불완전 — 채팅 베이스 레이어 필요

**문제**: PTT는 채널에 "지금 듣고 있는 사람"이 있어야 성립. 아무도 없으면 허공에 말하는 것.
- 채팅은 비동기라 상대 부재 시에도 메시지 남음
- AI 채팅은 AI가 항시 대기
- PTT는 항시 대기하는 누군가가 필요 → 타겟이 좁아짐 (같은 웹 서비스를 동시에 쓰며 근무하는 팀/조직)

**해결**: 채팅이 베이스 레이어, PTT가 그 위에 얹히는 구조
- 팀원 온라인 → PTT로 즉시 대화 (실시간, ambient awareness)
- 팀원 부재 → PTT가 음성 메시지로 자동 저장, 쓰레드에 쌓임
- 놓친 PTT → 쓰레드에서 재생
- 텍스트 선호 → 같은 쓰레드에 타이핑

**완전체 = PTT + 채팅 + 음성메시지** → "Intercom인데 음성 무전이 된다"

### PTT vs 음성 메시지(카톡) 차별점
- 음성 메시지: 녹음 → 전송 → 나중에 재생. 비동기.
- PTT: 버튼 누르는 순간 채널 전원에게 즉시 흘러감. **실시간 + ambient awareness**
- 팀이 동시 온라인일 때 → PTT 압도적. 동시 아닐 때 → 음성 메시지와 차이 없음
- 따라서 음성 메시지는 PTT의 **폴백**이지 핵심이 아님

### 서버 저장 & 보관 정책
- 웹 특성상 로컬 저장 의미 없음. 서버가 전부 보유
- 텍스트: 용량 미미, 보관 부담 없음
- 음성: Opus 기준 1분 ≈ 500KB. 팀 100개 × 30분/일 = 45GB/월. 무한 보관 불가
- **Slack 모델 참고**: 무료=90일, 유료=무제한. 보관주기 자체가 과금 포인트
- 보관 데이터는 oxrecd(녹화)와 자연스럽게 합류 가능

### 구현 난이도 평가
- SFU 미디어 엔진, PTT 파이프라인, WebRTC 시그널링 = **어려운 것. 이미 끝남**
- 프로세스 분리, 채팅, 위젯 패키징, 멀티테넌시, 관리 콘솔 = **일반 웹 개발. 설계만 완벽하면 공수 적음**
- oxrecd(녹화)만 미디어 엔진 영역이라 설계 깊이 필요
- 완성품이 먼저 있고 아이디어를 나중에 찾는 구조 → 돈이 안 태워짐. 안 먹히면 피벗, 엔진 그대로

## 11. 리스크

1. **"아무도 안 한" = "아무도 안 필요한"일 가능성** — Product Hunt/HN 무료 공개로 검증 (비용 ≈ 0)
2. **빅테크 진입** — Twilio/Agora가 PTT 위젯 만들면? → 선점+전환비용으로 방어
3. **1인 체제 병목** — "성공했을 때의 고민". 매출 나면 사람 쓸 수 있음
4. **Chrome 정책 변화** — WebRTC 제한 강화 시 영향. 단, Zoom/Meet도 같은 리스크
5. **타겟이 좁을 수 있음** — PTT는 팀 동시 근무 시나리오에서만 강점. 재택 유행도 끝남. 단, 위젯은 미끼 역할만 하면 됨
