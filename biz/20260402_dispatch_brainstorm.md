# 2026-04-02 사업화 브레인스토밍: 관제 + 웹 통신

## 결론

웹 PTT 디스패처가 빈틈. 통신 업체(ProPTT2, KT파워텔, BizPTT)는 영상 PTT + 관제 통합으로 가고 싶지만 네이티브라서 관제 웹에 못 들어감. OxLens는 웹 SDK로 유일하게 들어갈 수 있는 포지션.

## 국내 PTT 시장 현황

### 기존 통신 플레이어
- KT파워텔 RADGER: LTE 기반, 전용 단말기 + 앱, 영상 무전 3종 출시
- ProPTT2 (아이엠피티티): 영상 IP-PTT, SDK 제공, 레퍼런스 다수
- BizPTT (이수시스템): 앱 + PC 디스패처(네이티브)
- 사이버텔브릿지 EverytalkPTT
- Hytera HyTalk: 글로벌

### 공통점
- 전부 네이티브 앱 + 네이티브 PC 클라이언트
- 웹 디스패처 없음
- 에너지 방향: 음성 → 영상 PTT 확장, 관제 통합 지향
- 타겟: 공공기관, 제조, 건설, 유통, 물류

### 안전관제 업체 (통신 없는 관제)
- 세이콘, 다누시스(55개 지자체), 하이트론, 에이아이솔루션즈, 엔세이프, 카스웍스
- 전부 CCTV + IoT + AI 감지까지만
- 양방향 음성 통신 없음 (TTS 일방향 스피커, 푸시 알림 수준)
- PTT 업체와 파트너십도 거의 없음 — 아예 안 붙이고 있음

## 핵심 발견

### "관제 + 통신 통합"은 nice to have
- 현장은 이미 무전기 들고 다님. 관제와 분리되어 있어도 동작함
- must로 만들 법적 강제 근거 부족 (중대재해처벌법 증거 각도는 있으나 약함)

### 통신 업체의 딜레마
- 영상 PTT + 관제 통합으로 가고 싶음
- 근데 네이티브 아키텍처라서 관제 웹에 임베드 불가
- 관제 업체한테 "우리 앱 따로 띄우세요"밖에 못 함

### OxLens 차별화
- 웹 SDK → 관제 웹에 바로 들어갈 수 있는 유일한 선택지
- 관제 업체 입장: PTT 파트너 없이 자기 제품에 통신을 품을 수 있음

## 웹 디스패처 아키텍처 (초기 설계)

### 채널 구조
- 주채널 1개 (영상+오디오 크게) + 부채널 N개 (오디오만 모니터링)
- 부채널에서 PTT 올라오면 → 주채널로 자동 승격 가능 (웹이라 DOM 전환 즉각)

### PeerConnection 구조
- N채널 = 2N PC (pub+sub 모두 열어놓고 pub은 mute 대기)
- PTT 시 unmute만 하면 즉시 송신 (Room Join/PUBLISH_TRACKS 불필요)
- 8코어/32GB 기준 30채널(60PC) ~300MB — 충분

### 장치 전환 병목 없음
- 웹 브라우저: getUserMedia 이미 열려있고 모든 pub PC가 공유
- 부채널→주채널 전환 시 장치 구동 시간 = 0ms
- 네이티브 디스패처 대비 웹 디스패처의 강점

### STUN keep-alive 부하
- 60 PC × 30초마다 1회 = 초당 2패킷 (~112B/s) — 무시해도 됨
- SRTP 부재로 좀비 판단하지 않음 (ICE consent + WS heartbeat로 판단)

### 지령대 필수 기능
- 다중 채널 동시 모니터링 (채널별 볼륨, 뮤트)
- 채널별 PTT 송수신
- 영상 PTT 수신
- 접속자 목록/상태
- SOS 긴급 알림
- 통화 녹음/재생 (oxrecd 필요)
- GPS 위치추적 지도 (향후)
- TTS 방송 (향후)
- RoIP 게이트웨이 연동 (향후)

## RoIP 연동

### 가능 확인됨
- 업계가 RoIP + SIP/RTP + WebRTC 방향으로 이동 중 (REDCOM, Omnitronics 등)
- 기존 업체들은 SIP/RTP까지만 오고, WebRTC→웹 브라우저 구간이 빠져있음

### 예상 미디어 경로
```
아날로그 무전기 → RoIP GW → RTP/G.711 → Bridge Bot(트랜스코딩) → SRTP/Opus → SFU → 웹 브라우저
```

### 방향 2가 사업적으로 핵심
- 현장은 무전기 그대로, 관제 웹에서 기존 무전 채널에 붙는 구조
- 현장 장비 교체 없이 도입 가능 → 도입 장벽 최소

### Bridge Bot = OxLabs 봇 재활용
- OxLabs 봇이 이미 STUN+DTLS+SRTP+Opus publish 구현
- G.711 수신 + 트랜스코딩만 추가하면 됨
- SFU 코어 변경 불필요
