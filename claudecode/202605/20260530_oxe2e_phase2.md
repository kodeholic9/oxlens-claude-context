## §0 의무 점검 (시작 전 — 대량 읽기는 김과장이 직접 대조)

김대리는 골격(무엇을)만 준다. **거는 쪽 DC 스택의 정확한 구현은 "서버 받는 쪽 ↔ 봇 거는 쪽" 대조로 김과장이 도출**한다.

**(가) 설계 근거**
- `context/design/20260530_oxe2e_design.md` §5-②(봇 2단계), §6(판정 모델 — floor 확장).

**(나) DC 설계서 (프로토콜 근거)**
- `20260414_datachannel_design.md` — SCTP/DCEP 구조, unreliable 채널.
- `20260415_mbcp_datachannel_v2_design.md` — MBCP TLV 포맷 (TS 24.380 native).

**(다) 서버 DC 받는 쪽 — 봇은 이것의 역방향 (정합 기준)**
- `oxsfud/src/datachannel/mod.rs` — SCTP event loop + DCEP + MBCP dispatch.
- `oxsfud/src/datachannel/dcep.rs` — RFC 8832 DCEP 파서/빌더.
- `oxsfud/src/datachannel/mbcp_native.rs` — TS 24.380 TLV 빌더/파서 + 단위 테스트. **봇이 byte-level 정합해야 할 포맷의 단일 출처.**

**(라) oxrtc 현재 표면 (확장 대상)**
- `oxrtc/src/lib.rs`, `oxrtc/src/dtls_client.rs` — DTLS 후 demux 분기점(현재 SRTP만), `signal_client.rs`.

**(마) floor 서버 핸들러 (판정 근거)**
- `oxsfud/src/signaling/handler/floor_ops.rs` — MBCP binary 경로(DC bearer).
- `oxsfud/src/room/floor.rs` (FloorController) / `floor_broadcast.rs` (granted→user 단위 broadcast).

**(바) PROJECT_MASTER**
- "MBCP Floor Control" 섹션 — DCEP unreliable(maxRetransmits=0), 프레임 `[svc(1)+len(2)+payload]` svc=0x01, FIELD_DESTINATIONS count==1, viaRoom TLV, T101/T104.

**(사) sctp-proto 0.9 — 거는 쪽(client) association API.** ← 최대 리스크 지점.

**대조 산출**: 서버 받는 쪽 DC 흐름(SCTP accept→DCEP→MBCP dispatch) ↔ 봇 거는 쪽(association 시작→DCEP open→MBCP req) 매핑.

## §1 컨텍스트

- Phase 1로 **conf_basic(전환 없는 conference) PASS**. Phase 2 = **floor(PTT) 회귀**.
- v3에서 floor는 **DC-only MBCP**다. 봇이 floor를 *걸려면* oxrtc에 **클라 DC 스택**(SCTP→DCEP→MBCP)이 있어야 한다.
- 서버 oxsfud DC는 **받는 쪽이 검증됨**. 봇은 **거는 쪽 = 역방향** 구현.
- **최대 리스크 = SCTP association(sctp-proto 거는 쪽).** 정지점으로 끊는다.

---

## §2 결정된 사항

1. **oxrtc에 클라 DC 스택 추가 = 이 Phase의 본 작업** (설계 §4 정합).
   - ※ Phase 1의 *우발적* 발행 표면 보강(절차 모호했음)과 달리, **DC 추가는 처음부터 계획된 본 작업**이다. oxrtc를 건드리는 게 정상 범위 — 절차 혼선 없음.
2. **봇 = floor 거는 쪽**: MBCP **빌더**(FLOOR_REQUEST/RELEASE + FIELD_DESTINATIONS) + **파서**(Granted/Idle/Denied/Revoke/QueueInfo). (oxtapd는 파서만이지만 봇은 둘 다 필요.)
3. **충실도 = byte-level 정합**: MBCP TLV가 서버 `mbcp_native` 포맷과 바이트 단위로 일치. **상수 일치 ≠ wire 일치** — 실제 빌드 바이트를 서버 파서가 받는지 검증.
4. **서버 oxsfud DC 0 변경** (받는 쪽 검증됨).
5. **시나리오 ptt_rapid**: half-duplex 봇 여럿이 floor 회전.
6. **판정 = 약속↔이행의 floor 확장** (골격 — Phase F에서 §0(마) 대조로 구체화):
   - 약속 = FLOOR_MBCP Granted + TRACKS_UPDATE(vssrc).
   - 이행 = granted 봇의 audio가 **vssrc로** 다른 봇에 도착. 비발화 봇 송신은 fan-out 안 됨(half-duplex gating).

## §3 정지점 (3개 — SCTP 리스크로 잘게 끊음)

- **★정지점 1 — SCTP association 수립**: DTLS demux 분기 + sctp-proto 거는 쪽 association 성립. **최대 산.**
- **★정지점 2 — DCEP 채널**: OPEN/ACK → unreliable 채널 열림(서버가 채널 인지).
- **★정지점 3 — MBCP floor 왕복 + ptt_rapid PASS**: FLOOR_REQUEST→Granted 수신 + 판정 PASS.

각 정지점은 §6-1 워크플로(보고→검토→커밋)를 따른다.

## §4 단계별 작업

### Phase A — oxrtc DTLS demux 분기
- DTLS handshake 후 들어오는 패킷을 SRTP / SCTP로 분기(현재 SRTP만). DTLS app data = SCTP.

### Phase B — SCTP association (sctp-proto 거는 쪽) → ★정지점 1
- 거는 쪽 association 성립. **이게 최대 리스크.** sctp-proto 거는 쪽 API가 §0(사) 대조로 안 잡히면 착수 전 보고.
- **2회 실패 시 즉시 중단 + 방향 질문**(§6-4 엄격 적용).

### Phase C — DCEP (RFC 8832) → ★정지점 2
- DATA_CHANNEL_OPEN/ACK. unreliable 채널(maxRetransmits=0). 서버가 채널 인지하면 통과.

### Phase D — MBCP 빌더 + 파서
- 빌더: FLOOR_REQUEST + FIELD_DESTINATIONS(count==1). 파서: Granted/Idle/Denied/Revoke.
- **서버 `mbcp_native` 포맷과 byte-level 정합** — mbcp_native 단위 테스트가 기준.

### Phase E — oxe2e 봇 floor 로직
- 타임라인 floor 액션(actor, hold) → MBCP FLOOR_REQUEST → Granted 수신 → 발화 시작(half-duplex audio 송신) → hold 후 RELEASE.

### Phase F — 판정 확장 (floor phase별)
- granted 봇 audio가 vssrc로 다른 봇에 도착(이행), 비발화 봇 송신은 미도착(gating)을 검산.
- ※ conf_basic보다 복잡(floor 회전 + half-duplex + vssrc). §0(마) floor_ops/floor.rs 대조로 정확히. **막히면 2회 룰 적용 + 보고.**

### Phase G — ptt_rapid.toml + 실측 → ★정지점 3
- half-duplex 봇 2~3, floor 회전 타임라인. 실측 PASS.

## §5 변경 영향 범위

- **oxrtc**: DC 스택 추가 (본 작업). `dtls_client` demux 분기 + 신규 모듈(sctp/dcep/mbcp). **빌더+파서 분리** — oxtapd(미래)는 파서만 재사용하게 모듈 경계.
- **oxe2e**: floor 로직 + 판정 확장 + `scenarios/ptt_rapid.toml`.
- **불변**: `oxsfud` DC(받는 쪽) + 서버 전체 **0 변경**.

## §7 기각된 접근법

- **봇 reliable 채널** — MBCP는 unreliable(maxRetransmits=0). 서버 정합.
- **MBCP 자체 포맷 변경/재정렬** — 서버 `mbcp_native`(TS 24.380 native) 그대로. byte-level 정합 의무.
- **WS floor fallback** — Phase 2는 DC-only. (bearer=ws는 나중.)
- **full-ICE** — STUN binding만 유지(localhost+ICE-Lite).
- **서버 DC 손대기** — 받는 쪽 검증됨.
- **FIELD_DESTINATIONS count≥2** — Phase 1 서버는 count==1만(Pan-Floor 폐기). 봇도 count==1.
