## 본 세션 흐름

### 1. 부장님 통찰의 시발점

직전 세션 ("vssrc 모호성 발굴" 등) 의 후속으로, 부장님이 — 무전 vssrc 를 universal 컴파일 상수로 고정한 것이 single-room PTT 가정의 잔재이며, cross-room affiliate 환경에서 두 방의 동시 발언이 같은 vssrc 로 들어오면 NetEQ jitter buffer 가 깨진다 — 라고 진단.

김대리 1차 답변 (틀림): "(subscriber, room, kind)" 키 차원이라고 동의. 부장님 정정: "무전은 화자 단위가 될 수 없다, 방에 subs + audio/video 2개 불변". 김대리 2차 답변: 화자 단위 슬롯의 비합리성 인정 후 (subscriber, room, kind) 재정리 — 또 틀림 (Hall ≠ affiliate 분리).

부장님 통찰 (결정타): "Hall 은 거대룸 1나, 머릿속에서 지우고 affiliate 만 생각해라. 그럼 MCPTT 구조가 될 수밖에 없어, room-id, subs, kind 키. 이걸 해결하면 Hall 방이든 일반 방이든 상관 없을 것 같지 않아?".

→ **Hall 의 자료구조가 affiliate 의 부분집합 (N=1 케이스)**. 한 wrapper 로 통합. PROJECT_MASTER ② Hall 의 "HallSlotRewriter 신설" 이 사실은 별도 wrapper 가 아니라 PttRewriter 의 자연 확장이었음.

### 2. vssrc 키 차원 정정

김대리 추가 정정: vssrc 와 mid 는 키 차원이 다름.

| 자료 | 키 차원 | 이유 |
|---|---|---|
| **vssrc** | `(room_id, kind)` | wire 위 stream identity. 같은 방 모든 subscriber 가 같은 SSRC 받아도 됨 (LiveKit/mediasoup fan-out 정합) |
| **mid** | `(room_id, subscriber, kind)` | subscriber 입장 m-line index, subscriber 마다 다름 |

부장님 직관 본질 ("방마다 다른 vssrc") 은 정답. subscriber 차원만 빼면 자연 키.

### 3. 소스 확정 (Filesystem MCP 활용)

부장님 지적 ("소스 위치 접근 가능하잖아, 맥북인데") — 김대리가 PROJECT_MASTER + 대화 검색만 의존하던 한계를 지적. Filesystem MCP 로 직접 확인.

#### config.rs 발견

```rust
/// PTT audio universal SSRC ("PTT\xA1")
pub const PTT_AUDIO_SSRC: u32 = 0x50_54_54_A1;
pub const PTT_VIDEO_SSRC: u32 = 0x50_54_54_B1;
pub const PTT_AUDIO_RTX_SSRC: u32 = 0x50_54_54_A2;   // 미사용
```

**주석이 한계를 명시**:

→ Axiom 2 자체가 cross-room 도입 시 깨지는 가정.

### 4. 동작 차이 확인 (room_ops.rs)

부장님 질문: "방 1 join → 방 2 join → 방 1 leave 시 동작이 좀 달라야겠지, full / half에 따라서".

`room_ops.rs::handle_room_leave` 에 이미 한 줄 박혀 있음 (single-room 기준):

```rust
// PTT Unified Model: PTT slot 은 방 인프라로 영구 고정 —
// 개별 leave 가 subscriber 의 PTT m-line 에 remove 이벤트를 발송하면 안 된다.
let broadcast_remove_tracks: Vec<serde_json::Value> = remove_tracks.iter()
    .filter(|t| t.get("duplex").and_then(|v| v.as_str()) != Some("half"))
    .cloned()
    .collect();
```

이 한 줄이 두 트랙 종류의 본질 차이 표현:

| | full-duplex 트랙 | half-duplex (PTT slot) |
|---|---|---|
| 소유자 | 참가자(publisher) | 방(인프라) |
| 수명 | publisher 입장~퇴장 | 방 생성~소멸 |
| leave 시 broadcast | 다른 user SDP 에서 remove ✓ | 건드리지 않음 |

cross-room 으로 넓히면:

| 이벤트 | full-duplex | half-duplex (PTT slot) |
|--------|-------------|------------------------|
| 방 1 join | 방 1 user들 트랙 add (X SDP에) | 방 1 PTT slot 2개 add (vssrc_R1) |
| 방 2 join (affiliate) | 방 2 user들 트랙 추가 add | 방 2 PTT slot 2개 추가 add (vssrc_R2) |
| 방 1 leave | 방 1 user들 트랙 X SDP 에서 remove + 방 1 다른 user들에게 X 트랙 remove broadcast | **방 1 PTT slot 2개 X SDP 에서만 remove** — 방 1 다른 user들 영향 없음 |

**비대칭 핵심**: half-duplex slot 은 leaver SDP 에서만 정리. 방 다른 user들은 자기들끼리 PTT 계속.

## 기각된 후보

### Hall 별도 wrapper 신설 (PROJECT_MASTER ② 의 "HallSlotRewriter 신설")
