## 도출된 사실 정리

**S4** — admin sfu_metrics 메시지의 송신 주기는 3000 ms. 측정창 길이가 송신 주기 미만이면 before/after 가 같은 메시지로 캐치될 수 있고 그때 모든 카운터의 delta 가 0 이 된다. 측정창을 송신 주기 이상으로 설정하면 (3.5 s 측정 시) 동일 환경에서 `pub.rtp_in delta = 465`.

**S5** — 영상무전 시나리오에서 서버가 bob 에게 보내는 PTT virtual track 은 `track_id='ptt-audio'` / `'ptt-video'` (고정), `user_id` 가 null/missing. 클라이언트는 `applyTracksUpdate(action='add')` 에서 이를 그대로 `addParticipant(null)` 로 등록하므로 `remoteEndpoints` 의 key 가 `null`. `resolveOnTrack` 도 PTT 분기에서 `userId = null` 을 명시적으로 설정. 따라서 `bob.engine._currentRoom.remoteEndpoints.get('alice_vr')` 호출은 영상무전 시나리오에서 의도된 비매칭 (시험 expected 가 시나리오에 부적합).

## 부수 관측

- snapshot 메시지(top-level type='snapshot')의 키: `['rooms', 'ts', 'type', 'users']`. `pipeline` 키 없음.
- raw 50개 메시지 type 분류: `{snapshot: 17, hub_metrics: 17, sfu_metrics: 16}`
- 본 회차 시험 중 console errors: 0 건. console warnings: 32 건 (메시지 미캡처)

## 미시험 / 추후 조사

- `SfuMetrics::flush()` 를 호출하는 코드 위치 (lib.rs spawn 본문 미발견)
- sfu_metrics 메시지에 `pipeline` 키를 추가하는 코드 위치
- 서버 측 helpers.rs / track_ops.rs 에서 PTT virtual track 의 `t.user_id` 를 비우는 정확한 위치
- 영상무전 시나리오 PTT-aware 검증 메서드 정의 (`getAllSubscribePipes()` + `kind='video' && duplex='half'` 경로)

## 수정 권장 (다음 회차 시험 작성 시)

- 측정창 ≥ sfu_metrics 송신 주기 (3000 ms) 사용
- PTT 시나리오에서 영상 검증은 `_currentRoom.remoteEndpoints.get(userId)` 대신 `_currentRoom.getAllSubscribePipes().filter(p => p.kind==='video' && p.duplex==='half')` 사용
