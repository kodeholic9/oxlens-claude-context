## 개선 포인트 6가지

### 1. 컨텍스트 비용 의식하기

PROJECT_MASTER.md 600줄 이상 → 매 대화마다 전체 로딩 → 실제 작업 공간 압박.

**방법:**
- "변하지 않는 참조 정보"(미디어 파이프라인, RTCP, Simulcast 상세)를 별도 파일로 분리, SKILL은 "현재 진행 + 원칙 + 기각"만 (300줄 이하 이상적)
- 또는 세션 시작 시 "오늘 작업 영역" 먼저 선언 → AI가 관련 섹션에 집중

### 2. 세션 컨텍스트 "체급 조절"

현재 세션 파일 평균 200~400줄. 다음 세션에 필요한 건:
- 미해결 사항
- 기각/지침
- 설계 판단의 "왜"

**방법:** 세션 종료 시 "다음 세션용 요약" 5~10줄 작성. 전체 세션 파일은 아카이브 취급. 다음 세션에서 요약만 로딩.

### 3. 삽질 패턴 인식 — "증상 고치기 → 구조 바꾸기" 반복

| 사례 | 삽질 | 결론 |
|------|------|------|
| display:none | 5차 시도 | left:-9999px |
| PT 하드코딩 | 3월 발견 → 4월 잔재 | grep 한 번이면 끝 |
| m-line 누적 | assignMids 반복 수정 | 서버 할당 전환 |

**공통:** 증상 패치에 시간 쓰다 결국 구조 변경.

**방법:** 2차 실패 시 "업계 표준 해법이 뭐야?" 먼저 질문. 김대리 원칙 3번을 **부장님 본인에게도 적용**.

### 4. AI한테 "탐색"을 먼저 시키기

현재 패턴: 부장님이 방향 결정 → AI에게 설계/구현.

**더 빠른 패턴:** 새 문제 만나면 코드 보기 전에 먼저:
> "이 문제를 mediasoup, LiveKit, Janus는 어떻게 해결했어? 비교해줘."

부장님 직관 + 업계 선례 교차검증 = 최단 경로.

### 5. 코드 리뷰 적극 활용

현재: 코딩 → 테스트 → 버그 → 다시 세션.

**방법:** 코딩 후 한 번 더:
> "이 코드에서 놓쳤을 엣지 케이스 5개 찾아봐"

ensureHot 순서 버그, clear_speaker 이중 호출 → 코드 리뷰에서 잡을 수 있었음. 디버깅 세션 자체가 줄어듦.

### 6. 메모리를 전략적으로 쓰기

SKILL 파일과 중복되는 메모리 정리. 메모리는 **SKILL에 없지만 매 대화에 영향을 주는 것**에 집중.
예: RTCP Terminator 원칙 → PROJECT_MASTER.md에 이미 있으니 메모리에서 제거 가능.

## 부장님 반론 (2026-04-12)

> "빠른 길만이 정답은 아니다"

### 삽질의 가치
- **서버 주도 mid 할당은 cross-room 허들이 되었다** — "업계 정석"을 바로 따랐으면 이 trade-off를 체감 못 했을 것. 클라이언트 주도로 3세션 삽질하면서 SDP m-line 생명주기를 직접 뜯어본 경험이 서버 주도 설계의 약점까지 이해하게 함
- **경험 축적 자체가 목적** — 이건 납품 프로젝트가 아니라 1인 개발자의 역량 구축 과정. "최단 경로로 도착"보다 "길을 외우며 도착"이 장기적으로 옳을 수 있음

### 환경에 따른 전략
- **현재 220불 구독** — 토큰 제약 없는 환경에서는 시행착오에 투자할 여유가 있음
- **토큰이 귀해지면** — 그때 탐색 선행 전략으로 전환해야 함
- **결론: 절대 법칙이 아니라 환경 변수에 따른 전략 선택**

### 조언 적용 기준 (수정)
- 3번(2차 실패 시 선례 질문)과 4번(탐색 선행)은 **"빠르게 끝내야 할 때"** 적용
- **학습/탐구 목적일 때는 현행 패턴 유지** — 직접 부딪혀야 체득되는 것이 있다
- 5번(코드 리뷰)은 **환경과 무관하게 항상 유효** — 삽질 가치와 관계없이 버그 사전 차단은 순수 이득
