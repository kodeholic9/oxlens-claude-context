## 2. 근본 원인 패턴 분석

### 패턴 A: 상상 먼저, 사실 나중

| 상상 | 사실 |
|------|------|
| stream_map.clear()가 intent를 리셋한다 | 호출처 없음 (dead code) |
| admin 스냅샷 시점이 문제다 | 기능에 영향 없는 진단 도구 문제 |
| track.muted 타이밍 레이스다 | Chrome 내부 판정은 불확실, 의존 자체가 잘못 |
| 노트북 1대라서 화질이 낮다 | 4인 Conference는 문제없음 |

**교훈: 코드를 읽기 전에 결론을 내리지 마라.**

### 패턴 B: 동작하는 것과 비교하지 않음

부장님이 "오디오는 어떻게 동작하거니?"라고 물어보셔야 비로소 비교 시작.
**동작하는 경로(audio)와 안 되는 경로(video)를 비교하면 차이점 = 원인.**
이건 디버깅의 기본인데, 서버 로그/스냅샷 분석에 매몰되어 놓침.

### 패턴 C: 실행 파일을 확인하지 않음

- `demo/client/app.js`를 수정하고, `demo/scenarios/radio/app.js`가 실행되는 걸 몰랐음
- DEBUG 로그가 안 찍히는데도 "왜 안 찍히지?"로 멈추지 않고 다른 추측으로 넘어감
- **파일 수정 후 로그가 안 찍히면 → 첫 번째 의심은 "실행되는 파일이 맞는가"**

### 패턴 F: 스냅샷 분석 가이드 우선순위 무시

METRICS_GUIDE_FOR_AI.md에 명시된 분석 체인:
```
AGG LOG → TRACK IDENTITY → SDP STATE → Subscribe/Publish → Pipeline Stats → 코드 경로
```

실제로 한 것:
- TRACK IDENTITY 봄 → "admin 스냅샷 오래된 데이터"로 뛰어넘김 (우선순위 이탈)
- 서버 로그로 점프 (스냅샷 체인 중단)
- 스냅샷에서 "서버 릴레이 정상, 클라이언트 수신 정상" 확인되었는데도 서버 분석 반복
- **"스냅샷에서 코드 경로까지 도달"의 "코드 경로" = 클라이언트 코드 포함**인데, 서버 코드만 뒤졌음

**교훈**: 스냅샷에서 "서버 정상" 확인되면 체인을 **클라이언트 코드 경로**로 이어가야 함. 서버 로그를 더 파는 것이 아님.

### 패턴 D: 스냅샷 분석에 과도한 시간 투자

스냅샷 3개를 분석하면서 서버/네트워크/SDP를 반복 확인.
하지만 스냅샷은 "서버 릴레이 정상"이라는 결론만 줄 수 있었음.
**"서버가 정상이면 클라이언트다"로 빠르게 전환했어야 함.**

### 패턴 E: 노트북 환경 핑계

3번 이상 "노트북 1대라서"를 언급.
부장님이 매번 차단하셨음. **환경 탓은 데이터 근거 없이 금지.**

## 3. 김대리 행동 원칙 추가 제안

### ⭐ 스냅샷 분석 가이드 우선순위 준수
- METRICS_GUIDE_FOR_AI.md의 분석 체인을 끝까지 따라간다
- "서버 정상" 확인 시 체인의 "코드 경로" = 클라이언트 코드 경로로 이어감
- 서버 로그로 점프하지 않음

### ⭐ 동작하는 것과 비교하라 (A vs B 디버깅)
- "X가 안 된다" → "비슷한 Y는 되는가?" → "X와 Y의 코드 차이점이 원인"
- audio가 되면 audio 코드 경로와 video 코드 경로를 비교
- Conference에서 되면 Conference 코드와 PTT 코드를 비교

### ⭐ 수정 후 반영 확인
- 로그가 안 찍히면 → "실행되는 파일이 맞는가?" 먼저 확인
- HTML이 import하는 JS 경로를 확인할 것
- 시나리오별 별도 JS 파일 존재 인지

### ⭐ 서버 정상 → 클라이언트로 즉시 전환
- 스냅샷에서 "릴레이 정상, 수신 정상" 확인되면 서버 분석 중단
- 브라우저 콘솔 → 코드 경로 추적으로 전환

### ⭐ 노트북 핑계 금지
- "환경 탓"은 결론이 아님. 데이터로 근거를 제시해야 함
- 같은 환경에서 다른 모드가 정상이면 환경 문제가 아님

│   │   ├── index.html
