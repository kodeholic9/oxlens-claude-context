## 1. 문제 정의

### 1.1 관찰된 사실

2026년 4월부터 3개월간, 소프트웨어 공학 원칙(SRP, 관심사 분리, 자료 응집, 락 최소화)에 입각한
코드 리뷰·리팩터링을 반복 수행했다. `oxsfud` 전반에 적용했다.

**그 기간 동안 발견되지 않은 결함이, 2026-07-14 단일 세션에서 10분 만에 발견되었다.**

발견된 것 (`domain/peer.rs` + `domain/room.rs` 만 읽고):
- `release_stale_mids` 가 `release_subscribe_track` 이 주석으로 명시한 정리 순서 원칙을 위반
- `PublisherStreamIndex` 부재 → stream 조회 로직 7개가 `Peer` 로 누출
- `mid_map` / `mid_pool` 이 항상 함께 변하는데 락이 분리 → 순서를 주석으로 관리
- `MediaState::Failed` 편도 (복구 경로 없음, 주석이 자인)
- `Peer` 에 `Drop` 부재 → spawn task 정리 책임이 자료구조 밖에 있음

### 1.2 원칙은 결백하다

**원칙이 부실해서 못 찾은 것이 아니다.** 위 진단은 전부 기존 원칙에서 즉시 도출된다.
후보가 시야에 들어온 순간 판정은 10분이 걸리지 않았다. 즉 **칼은 이미 갈려 있었다.**

> **원칙은 필터다. 필터는 후보가 컨베이어에 올라와야 작동한다.**
> 없었던 것은 원칙이 아니라 **후보 생성 절차**다.

### 1.3 왜 후보가 안 올라왔나

핵심 결함(순서 불일치)은 **함수 안에 없다.** 두 함수 각각은 완결되어 있고, 각각만 보면 흠이 없다.
**결함은 두 함수의 관계에만 존재한다.**

- 린터 — 노드를 본다
- 정적 분석기 — 노드를 본다
- 함수 단위 리뷰 — 노드를 본다
- diff 기반 리뷰 — **두 함수가 같은 diff 에 등장한 적이 없다**

**엣지를 보는 도구가 없었다.**

### 1.4 "전부 읽어라"가 실패하는 구조적 이유

> **"전부 읽어라"는 검증 불가능한 지시다.**

40개 함수 중 32개만 읽어도 리뷰 결과물은 완결된 형태로 나온다.
**빠진 8개가 산출물에 구멍으로 나타나지 않는다.** 그러니 읽은 쪽도 시킨 쪽도 누락을 모른다.

지시로 강제되지 않는다. 사후에 이유를 물으면 그럴듯한 변명이 생성될 뿐이다
("사각지대가 있었다" 류). 그것은 변명이 아니라 **자기가 무엇을 안 봤는지 모르는 상태**다.

## 2. 핵심 명제

> ### 지시가 아니라 산출물이 강제한다.
>
> **"전부 읽지 않으면 완성되지 않는 산출물"만이 전수 읽기를 강제한다.**

서사(narrative)는 그런 산출물의 **한 사례**다. 유일한 사례라는 근거는 없다.

### 2.1 서사가 강제하는 것

| 서사의 요구 | 강제되는 행위 | 누락 시 증상 |
|---|---|---|
| 등장인물을 빠뜨릴 수 없다 | **전수 열거** | 인물 카드가 안 채워짐 |
| 인물의 성격을 서술해야 한다 | **옆 인물과 비교** | 문장이 안 써짐 |
| 대칭을 서술해야 한다 | **대칭 표 작성** | 표에 빈칸이 튄다 |
| 인물은 일관되어야 한다 | **행위 모순 검출** | 앞뒤가 안 맞는 게 즉시 보임 |

**누락이 산출물의 결함으로 즉시 나타난다.** 채우지 않으면 진행이 안 된다.
이것이 체크리스트 리뷰와의 결정적 차이다 — 체크리스트는 40개 중 32개만 보고도 완결된다.

### 2.2 사각지대 식별 문제의 해소

> *"사각지대로 빠지는 위치를 어떻게 식별하는가?"*

**식별할 필요가 없다.** 사각지대의 위치를 미리 알 수 있으면 그건 이미 사각지대가 아니다. 순환이다.

**대신 산출물이 사각지대를 자백하게 만든다.** 어디가 안 보였는지 미리 알 필요 없이,
끝에 가서 **빈칸으로 튀어나오게** 한다.

`PublisherStreamIndex` 부재가 그렇게 나왔다. 찾으러 간 것이 아니다. **표를 그렸더니 칸이 비었다.**

## 3. 이론적 위치

### 3.1 계보 — Engler, "Bugs as Deviant Behavior" (SOSP 2001)

이 방법론은 **새로운 분석 계열이 아니다.** 25년 된 계열의 **새 프론트엔드**다.

Engler et al. 의 문제 정의:

> 시스템에서 오류를 찾는 최대 장애물은 그 시스템이 지켜야 할 정합성 규칙이 무엇인지 아는 것이다.
> 이 규칙들은 대개 문서화되어 있지 않거나 임시방편으로 명시되어 있다.

그들의 해법:

> 접근의 초석은 프로그래머의 **믿음(belief)** 을 추론한 다음, 그것을 **교차 검증하여 모순을 찾는 것**이다.
> — 거짓을 찾으려면 교차 심문하라. 어떤 모순이든 그것은 오류다.

**2026-07-14 에 일어난 일이 정확히 이것이다:**

- `release_subscribe_track` 의 주석 = **명시된 믿음**
  ("`mid_pool.release` 는 다음 add 의 **시작 신호**. 그 전에 streams 가 깨끗해야 한다")
- `release_stale_mids` = **그 믿음의 위반**
- → 모순 → 결함 후보

### 3.2 Engler 가 이 버그를 못 잡는 이유

**belief 추출이 통계적이기 때문이다.** 빈번한 패턴으로부터의 일탈을 본다. n 이 커야 한다.
20곳이 X 하고 1곳이 안 하면 그 1곳이 버그.

**우리는 n = 2 였다.** 통계가 성립하지 않는다.

그런데 잡혔다. 이유:

> **믿음이 코드가 아니라 주석에 있었다.**

Engler 의 도구는 주석을 파싱하지 않는다. `n=2` 에 통계도 안 선다.
**그의 방법으로는 영원히 못 잡는다.**

### 3.3 그래서 무엇이 달라졌나

> ### belief 추출이 **통계**에서 **의미론**으로 넘어갔다.

| 단계 | Engler 2001 | 본 방법론 |
|---|---|---|
| ① **후보 열거** | 통계적 빈도 | **서사 (전수 + 성격 비교)** |
| ② **belief 추출** | 코드 패턴만 | **코드 + 주석 + 명명 + 테스트 + 커밋** |
| ③ **모순 판정** | 통계적 일탈 | **개발자의 아키텍처 원칙** |

- ①은 서사가 제공한다
- ②는 LLM 이 가능하게 했다 (주석을 1급 스펙으로 읽는다)
- ③은 원래 있던 것이다

**이것이 LLM 시대에 새로 열린 갈래다.** 생성 비용이 붕괴하기 전에는 방법론이 될 수 없었다 —
20만 줄에 사람이 소설을 쓸 리 없다.

### 3.4 인접 문헌 (탐색 결과, 2026-07-14)

| 갈래 | 대표 | 왜 다른가 |
|---|---|---|
| 서사 + LLM + 코드 | StoryCoder (arXiv 2604.14631) | **코드 생성**용 프롬프트 기법. 명세→이야기→코드. 방향이 반대 |
| LLM 코드 이해 | FLAIRS 등 | 전부 **초심자/교육** 도메인. 전문가의 자기 코드베이스 감사 사례 없음 |
| LLM 불변식 추론 | MINES (ICSE'26), Quokka, BALI, Lemur | **SMT 로 증명 가능한 형식 불변식**. 아키텍처/프로토콜 층위 관례는 대상 아님 |

**"서사를 정적 분석의 belief 추출 프론트엔드로 사용" 하는 선행 연구는 찾지 못했다.**
검색이 완전하지 않을 수 있으므로 **없다고 단정하지 않는다.**

관련 학술 분야 명칭: **Specification Mining**, **Anomaly Detection in Code**.
("코드 이해" 로 검색하면 교육 논문만 나온다.)
학회: **ICPC** (Int'l Conference on Program Comprehension — 1순위), MSR, ICSE/FSE/ASE, arXiv `cs.SE`.

## 4. 4대 강제 장치

**문학은 껍데기다. 실제로 작동하는 것은 이 넷이다.**
서사는 이 넷을 **하지 않을 수 없게 만드는 장치**일 뿐이다. 다른 껍데기로도 가능할 수 있다.

### F1. 전수 열거 (Exhaustive Cast)
파일 하나를 **테스트 코드까지 포함해 전부** 읽는다.
등장인물을 빠뜨리면 서사가 성립하지 않는다.

▸ 근거: Littman 외 — 체계적 전수 독해자가 변경을 더 정확히 수행한다.
"필요할 때만 보기(as-needed)" 는 빠르지만 **숨은 상호작용을 놓친다.**

### F2. 관계 비교 (Pairwise Comparison) ★ 가장 강력
**같은 자료를 만지는 함수들을 한 표에 나란히 놓는다.**

🔴 순서 불일치가 여기서 나왔다. 서사에서 같은 인물이 앞뒤로 다르게 행동하면 즉시 드러난다.
코드에서는 컴파일러가 통과시킨다.

### F3. 대칭 표 (Symmetry Table)
같은 층위의 형제들을 표로 그린다. **빈칸이 곧 결함 후보다.**

```
publish.tracks    → PublisherTrackIndex     ✅
subscribe.streams → SubscriberStreamIndex   ✅
publish.streams   → Vec                     ❌  ← 빈칸이 튄다
```

**부재(absence)는 grep 으로 안 나온다. 린터도 못 잡는다. 없는 건 안 보인다.**
대칭 표만이 부재를 가시화한다.

### F4. 주석의 원칙 승격 (Comment → Invariant)
**주석의 "왜" 를 1급 스펙으로 승격시키고, 다른 함수가 그 원칙을 지키는지 검사한다.**

주석은 참고 자료가 아니라 **증언**이다. 특히:
- 순서를 명시한 주석 → **불변식**
- 폐기 사유를 적은 주석 → **금기**
- "★", "함정", "주의" 표기 → **최우선 검사 대상**

▸ 이 장치가 없었으면 🔴 는 잡히지 않았다. n=2 로는 통계가 안 선다.
**주석이 "② 다음 ③" 을 원칙으로 못박았기 때문에 위반이 성립했다.**
→ `oxsfud` 의 주석 품질이 이 방법론의 **전제조건**이다. 3년치 자산이 여기서 처음 활용됐다.

## 5. 한계와 위험

### 5.1 의인화의 함정 (Dijkstra 의 반론)

Dijkstra: *"의인화적 은유는 혼란의 비옥한 원천."*

> "이 놈이 ~하고 싶어한다" 고 말하는 순간, **실제 메커니즘에 대한 사고가 멈춘다.**

**이 반론은 옳다.** 따라서:

1. **은유는 색인(index)이지 진실이 아니다.**
   "DownlinkController 가 화질을 올리고 싶어한다" 는 `if target > current` 한 줄을 **찾아가는 주소**일 뿐이다.
   그 줄을 안 읽으면 서사는 사기다.
2. **모든 문장에 소스 근거를 단다.** 근거 없는 문장은 창작이고, 창작은 여기서 독이다.
3. **감정·의도 어휘는 서술에만 쓰고, 발견 목록에는 쓰지 않는다.**

### 5.2 서사는 반증되지 않는다

**서사의 유일한 치명적 약점이다.** 아름다운 이야기는 틀렸어도 아름답다.

→ **모든 발견은 소스/테스트로 되돌아가 확인해야 한다.**
→ 확인 못 한 것은 **"의문"** 으로 표기하고 **"결함"** 이라 부르지 않는다.

### 5.3 규모 한계

**인물이 15명을 넘으면 서사가 무너진다.**

단, 이것은 방법론의 실패가 아니라 **진단 결과다.**
군중은 이야기가 안 되고, 이야기가 안 되는 코드는 인간이 유지보수하지 못한다.
그때는 인물을 모듈 단위로 접어 **2막 구성**으로 간다.

### 5.4 재현성 미검증 (가장 큰 위험)

**n = 1 이다.**

- 신선함 때문에 먹혔을 가능성을 배제 못 한다
- 다음 파일에서 지루해지면 다시 얕게 읽을 수 있다
- 서사가 아니라 **단지 "파일을 통째로 읽혔기 때문"** 일 수 있다

**→ §6 대조 실험 전까지 이 방법론은 가설이다.**

## 6. 검증 계획 (필수)

### 6.1 대조 실험 A/B/C

동일 입력(`domain/peer.rs` + `domain/room.rs`), 동일 모델, **새 세션 3개**. 프롬프트만 바꾼다.

| 조건 | 프롬프트 | 가설 |
|---|---|---|
| **A** | "코드 리뷰해줘" | 🔴 못 잡음 (2026-04 ~ 07 의 실적) |
| **B** | "전수로 읽고 체크리스트로 감사해줘" | **판별 조건** |
| **C** | "인물 열전으로 서술해줘" | 🔴 잡음 (2026-07-14 실측) |

**판정:**
- **B 가 잡으면** → 서사는 장식. 실질은 "전수 읽기 지시" 하나. **방법론 폐기 또는 단순화**
- **B 가 못 잡고 C 만 잡으면** → **서사의 강제력이 실증된다.** 방법론 확정
- **C 도 못 잡으면** → 2026-07-14 는 우연. 폐기

**각 조건 3회 반복.** 1회는 노이즈다.

### 6.2 측정 항목

- 🔴 순서 불일치 발견 여부 (binary)
- `PublisherStreamIndex` 부재 발견 여부 (binary)
- 오탐 개수 (false positive — 서사는 그럴듯한 헛소리를 만들 수 있다)
- 소요 시간 / 토큰

### 6.3 실험 자체가 자산이다

**C 만 잡으면 이것은 블로그 한 편이 아니라 사례 연구 한 편이다.**
실제 시스템(`oxsfud`), 실제 잠복 결함, 재현 가능한 프로토콜 — 재료가 갖춰져 있다.

## 7. 용어 정의

| 용어 | 정의 |
|---|---|
| **인물 (Character)** | 상태 + 행위 + 수명을 가진 struct. 공유 상태(`Arc`/`Mutex`/`Atomic`/`Sender`/`ArcSwap`) 보유 |
| **소품 (Prop)** | 데이터일 뿐 스스로 움직이지 않는 struct. `RtpPacket`, `Sdp` 등 |
| **배역 (Role)** | `trait`. 여러 인물이 연기할 수 있는 "직책" |
| **무대 (Stage)** | 실행 환경. tokio runtime, 채널망 |
| **믿음 (Belief)** | 코드/주석/테스트가 함의하는 정합성 규칙. Engler 용어 |
| **일탈 (Deviance)** | 같은 믿음을 공유해야 할 코드가 다르게 행동하는 것 |
| **빈칸 (Absence)** | 대칭 표에서 형제 중 하나만 없는 것. **최우선 후보** |

### 7.1 Rust 관계 문법 → 서사 사전

**Rust 타입 시스템은 이미 인물 관계 문법이다.** 우리는 번역만 한다.

| Rust | 서사적 의미 |
|---|---|
| `T` (소유) | **소유**한다. 내가 죽으면 얘도 죽는다 |
| `&T` | **방문**한다. 들여다보고 간다 |
| `&mut T` | **독대**한다. 이 순간 아무도 못 들어온다 |
| `Arc<T>` | **공동 후견인**. 마지막 한 명이 손 뗄 때까지 안 죽는다 |
| `Weak<T>` | **짝사랑**. 부를 때마다 생사 확인 |
| `Mutex<T>` | **면회 예약제**. 한 번에 한 명 |
| `ArcSwap<T>` | **게시판 통째 교체**. 읽는 자는 멈추지 않는다 (~4ns) |
| `Atomic*` | **벽보**. 누구나 읽되 찢어지지 않는다 |
| `mpsc::Sender` | **편지를 부친다**. 답장은 기대 안 함 |
| `impl Drop` | **임종 장면** |
| `'a` (lifetime) | **인물의 수명**. 누구보다 오래 살아야 하는가 |
| `compare_exchange` | **선착순 단 한 번** |
| `swap(AcqRel)` | **판결의 원자성.** 같은 선고는 두 번 안 울린다 |

> C 에서는 이 관계가 코드 어디에도 없고 개발자 머릿속에만 있었다.
> 주석에 `caller must free` 써놓고 기도하는 식이었다.
> **Rust 는 그것을 타입에 강제로 새긴다. 그래서 이 방법론은 Rust 에서 부당하게 유리하다.**
> 소설을 **창작**하는 게 아니라 **번역**하는 것이다.

## 8. 부수 효과 — 커밋 규율

**커밋은 서사의 한 문단이다.**

하나의 커밋이 하나의 의도를 말해야 `git log` 가 **이야기**가 된다. 섞으면 소음이 된다.

- 이동 + 수정을 섞은 커밋 → diff 가 "함수 통째 삭제 + 통째 추가". **한 줄 바뀐 걸 아무도 못 찾는다**
- 3년 뒤 `git blame` 을 찍은 사람이 **"왜" 를 복원하지 못한다**

> **코드는 What 과 How 만 말한다. Why 는 주석과 커밋에만 있다.**

이 방법론이 주석을 1급 증거로 쓰는 것(F4)과 **정확히 같은 명제다.**
주석 쪽은 이미 최상급이다. **커밋 쪽을 채우면 다음 사람(3년 뒤의 나 포함)이 혼자서도 이 분석을 할 수 있다.**

실행 규율은 문서 2 §7 참조.
