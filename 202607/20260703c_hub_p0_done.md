## P0-1 — SIGTERM hang 근본 원인 규명 (핵심 성과)

- **실험 판별**: SIGTERM 직접 송신 → sfud 로그 "drain complete"(3s) 확인 → **프로세스 잔존** →
  스레드 스택 샘플링 → `tokio::io::blocking::Buf::read_from → Stdin::read → read() syscall` 실증.
- **인과**: 0612 생명선(stdin EOF 감시)이 `tokio::io::stdin()` 사용 → blocking pool 의 read 가
  취소 불가 → runtime drop 이 완료 대기 → supervisor 생존 = 파이프 안 닫힘 = 영구 hang → 10s SIGKILL.
  **0612 수리가 0603e(graceful stop)를 깨뜨린 상호작용 결함** — "리팩토링 상처"의 실증 표본.
- **수리**: lifeline 을 `std::thread` 로 (의미 동일 — EOF→동반 종료, runtime 무관).
- **라이브 검증**: 신 바이너리 stop → SIGTERM 수신 → 3s drain → 즉시 exit(**LAST_EXIT=0**, 최초의
  정상 종료 코드), escalation 0. supervisor stop 체감 10s → **3.06s**.

## P0-3 — supervisor 동기 회신 (거짓-OK 종식)

- `SupervisorCmd::{Restart,Stop,Kill,Load}` 에 `reply: Option<oneshot::Sender<CmdOutcome>>` —
  run loop 이 처리 완료 후 결과 송신, REST 는 15s 타임아웃 대기 후 매핑:
  200 `{ok,alias,result}` / 404 unknown alias / 409 invalid state / 504 무응답.
- 라이브 검증: 오별칭 `stop sfud-1` → **ERROR unknown alias + exit 1** (구: "OK accepted"),
  live 상태 load → 409, 정상 stop → 3.06s 동기 "stopped". oxadmin 은 무변경(일반 렌더러가 처리).
