|---|---|

| **R4 불변식** | INVARIANTS 로 승격 가능한 원칙을 **위반**하고 있음 |
| **R5 핫패스** | 발견 지점이 **RTP/RTCP 핫패스** 또는 **fan-out 경로** |

**강등 규칙:**
- 앵커가 주석 하나뿐이고 코드 근거가 없음 → ❔

### Phase D — 모순 검출

**두 편이 같은 인물/자료를 다르게 서술했다면 하나는 틀렸다.**

검사 대상:
- 같은 struct 의 **책임 서술**이 편마다 다른가
- 같은 함수의 **호출자**가 편마다 다르게 적혔는가
- 같은 자료의 **소유자**가 편마다 다른가
- 같은 락의 **순서**가 편마다 다르게 적혔는가

**모순 발견 시**: 소스로 실측하여 어느 쪽이 맞는지 판정. **양쪽 다 틀릴 수도 있다.**

### Phase F — 🔴 동시성 판정 ★★★ (P0 게이트)

**단일 질문**:

> **같은 `Peer` 에 대해 `SubscribeContext::release_stale_mids` 와
> `Peer::release_subscribe_track` 이 동시 실행될 수 있는가?**

**배경** (세션 01 발견):

| 함수 | 정리 순서 |
|---|---|
| `release_subscribe_track` | ① mid_map → ② streams → ③ mid_pool |
| `release_stale_mids` | ① mid_map → ③ mid_pool → **(락 전부 해제)** → ② streams |

`release_subscribe_track` 주석이 **원칙을 명시**한다:
> *"mid_pool.release — 다음 add 의 **시작 신호**. 그 전에 streams 깨끗해야 `add_subscriber_stream`
> idempotent 분기에 옛 publisher_ref 잔재 안 끼어듦"*

`release_stale_mids` 는 **③을 ②보다 먼저 낸다.** 그 사이 락이 다 풀린다.

**조사 항목:**

1. `release_stale_mids` **호출부 전수** (`rg -n 'release_stale_mids'`)
   → `collect_subscribe_tracks` 인가? 다른 곳도 있는가?
2. `release_subscribe_track` **호출부 전수**
   → `helpers.rs` / `tasks.rs` / `room_ops.rs` — "옛 3중복" 이 정말 다 흡수됐는가?
3. **두 경로가 같은 peer 에 대해 동시 진입 가능한가?**
   - gRPC dispatch 는 요청마다 별 task 인가?
   - 같은 user 의 두 요청(예: ROOM_SYNC + ROOM_LEAVE)이 병렬 처리되는가?
   - **peer 단위 직렬화 장치(SerialLock 류)가 서버에 있는가?** ← 핵심
4. `add_subscriber_stream` 의 멱등 분기가 그 창(window)에서 옛 stream 을 반환할 수 있는가?

**판정 3택:**

| 판정 | 의미 | 후속 |
|---|---|---|
| **직렬 확정** | 동시 진입 불가 | 🔴 → 🟢 강등. 순서 통일은 **위생 작업** |
| **동시 가능** | window 존재 | **🔴 확정. P0** → 재현 테스트 작성 |
| **미확정** | 판단 근거 부족 | 부장님께 질문. **추측 금지** |

**Phase F 산출**: 판정 + 근거(호출 경로 그래프) + **★ 정지점 2 보고**
