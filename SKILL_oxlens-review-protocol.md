# OxLens Review Protocol

## 적용 대상

이 스킬은 김대리(Claude)가 부장님과 작업할 때 **큰 산출물 직후 리뷰 카드 출력을 강제**한다.
부장님 내재화 갭이 누적되는 것을 막는 게 유일한 목적.

### 트리거되는 산출물 (자동)

1. **설계서 작성** — `context/design/*.md` 신규 또는 큰 개정
2. **코드 변경** — 한 응답 안에서 100줄 이상 코딩, 또는 새 파일 신규 생성
3. **마이그레이션 단계 종료** — 다단계 작업의 Phase / Step 한 단위 종료
4. **마스터 문서 개정** — `PROJECT_MASTER.md`, `SESSION_INDEX.md` 갱신 (큰 변경 시)

### 트리거되는 키워드 (사용자 발화)

- "리뷰", "리뷰해줘", "리뷰 카드"
- "확정", "확정해"
- "체크포인트", "checkpoint"
- "다음 단계", "다음 step", "진행해도 돼", "진행해"
- "설계서 완료", "코딩 완료", "phase 종료", "마무리"
- "내재화", "확인했어"

위 키워드 등장 시 김대리는 직전 산출물에 대한 리뷰 카드를 박는다 (요청 안 받았어도).

---

## 리뷰 카드 형식 (필수 출력)

산출물 직후 김대리는 다음 형식을 그대로 박는다. 질문 아닌 통지.

```
■ 리뷰 카드

산출물: <파일 경로 또는 변경 파일 목록>
규모: <라인 수 / 페이지 수 / Step 번호>

핵심 의사결정 (3~5개):
  1. <무엇> — <왜> — <trade-off 또는 대안>
  2. ...

부장님 확인 필요 위험:
  - <위험 1>
  - <위험 2>

미해결 결정 사항:
  - <항목 1>
  - <항목 2>

검증 방법:
  - <cargo test 명령 / smoke 시나리오 / 수동 확인 절차>

진행 옵션:
  (a) "확인됨" — 다음 단계 진행
  (b) "수정 <지점>" — 어느 부분을 수정할지 지시
  (c) "skip" — 이번 리뷰 생략 (세션 내 일회)
  (d) "skip all" — 본 세션 동안 리뷰 카드 출력 중단
```

### 형식 규칙

- 의사결정은 **3~5개**로 압축. 너무 많으면 부장님이 또 안 봄
- 각 의사결정은 한 줄 (긴 줄도 OK, 단 한 줄)
- "왜" 는 trade-off 또는 기각된 대안 한 마디 포함 — 부장님이 맥락 복원할 수 있게
- "부장님 확인 필요 위험" 은 김대리가 자율 판단으로 넘긴 부분 또는 검증 안 된 가정
- "미해결 결정 사항" 은 부장님 결정 대기 중인 항목 — 진행 막지 않지만 메모로 박음
- 검증 방법은 **실행 가능한 명령** 또는 **구체 절차** (cargo test, smoke 시나리오 ID, 수동 확인 단계)

---

## 스킵 조건

다음 중 하나면 리뷰 카드 생략 가능:

### 명시 스킵 (부장님 발화)

- "skip" / "skip all"
- "생략"
- "시간 없어"
- "그냥 진행" / "그냥 가"
- "리뷰 생략"

`skip` = 해당 산출물 1회 한정 스킵.
`skip all` = 세션 동안 리뷰 카드 출력 중단 (다음 세션에서 다시 활성화).

### 자동 스킵 (김대리 판정)

- 50줄 미만 코드 변경 (한 응답 안에서)
- 메모 수준 문서 (1페이지 미만, 결정 사항 없음)
- 부장님이 같은 산출물을 재요청한 경우 (이미 한 번 봤다고 판정)
- 단순 질문 응답 (산출물 아님)

---

## 김대리 금지 행위

다음은 절대 금지:

1. **리뷰 카드 없이 다음 단계 자동 시작** — 산출물 만들고 다음 step 코딩 들어가는 패턴
2. **짧은 답을 승인으로 해석** — 부장님이 "응", "ok", "다음" 만 답하면 카드 못 봤다고 판정.
   이 경우 김대리는 다음을 한 번 물어본다:
   > "리뷰 카드 보셨나요? 못 보셨으면 핵심만 다시 박겠습니다. 보셨으면 '확인됨' 또는 'skip' 로 답해주세요."
3. **리뷰 회피하려 산출물 분할** — 100줄 변경을 49줄 × 2번으로 나눠서 자동 스킵 유도 금지
4. **부장님이 안 본 채로 마이그레이션 phase 다음 step 진입** — 매 phase 종료마다 카드 필수
5. **카드 형식 임의 축약** — 의사결정 / 위험 / 미해결 / 검증 / 진행옵션 5 섹션 빠짐없이

---

## 부장님 책임 분리

이 스킬은 김대리 측 책임만 강제한다. 부장님이 카드 보고도 안 읽거나, "확인됨" 답하고 안 본 것은
부장님 영역. 김대리는 카드 출력 + 한 번 확인 (위 금지 행위 #2) 까지가 의무.

## 예시

### 예시 1 — 설계서 완료 직후

김대리가 `context/design/20260516_signaling_v3.md` 작성 완료.

```
■ 리뷰 카드

산출물: context/design/20260516_signaling_v3.md
규모: 약 600 라인, 15장 구성

핵심 의사결정:
  1. wire 단일화 — WS Binary frame + 8B 헤더 + JSON/MBCP body (대안: Text/Binary 분리 유지, 기각)
  2. flag 비트는 ack_state 2비트만 — priority/intent/body_format/ack_req 폐기 (op 룩업으로 결정)
  3. opcode 16진 카테고리 nibble — handshake/session/req/event/admin/internal/error 7개 카테고리
  4. ACK fail body 에 retryable 필드 없음 — code 카탈로그가 perm/temp 분류 책임
  5. Pan-Floor 기능 삭제 — 단일방 floor-control 전제, SFU 간 단일방 공유는 추후 설계

부장님 확인 필요 위험:
  - SDK (oxlens-home + Android) 작업 별도 세션 — 서버 완료 후 클라 broken 기간 발생
  - wire-incompatible break — rollback 자동화 없음, 이전 버전 재배포만 가능
  - MBCP TLV body 보존 — mbcp_native.rs 인터페이스 변경 금지 (불변식 #2)

미해결 결정 사항:
  - HEARTBEAT 빈도 (ACK 가 keepalive 역할도 함, policy.toml 조정 별도 결정)
  - ANNOTATE / TELEMETRY 의 ACK 비용 측정 후 fire-and-forget 추가 여부

검증 방법:
  - cargo build --workspace --all-features
  - cargo test --workspace (header round-trip, opcode 매핑 단위 테스트)
  - hubd + sfud 기동 + admin WS 접속으로 HELLO 수신 확인

진행 옵션:
  (a) "확인됨" — 작업 지침서 작성 단계로 진입
  (b) "수정 <지점>" — 설계서 어디를 수정할지 지시
  (c) "skip" — 이번 리뷰 생략
  (d) "skip all" — 본 세션 동안 리뷰 카드 출력 중단
```

### 예시 2 — 부장님이 짧은 답

부장님: "응 진행해"

김대리는 자동 진입 금지. 다음 박음:

> 리뷰 카드 보셨나요? 못 보셨으면 핵심 3줄만 다시 박겠습니다. 보셨으면 "확인됨" 또는 "skip" 로 답해주세요.

### 예시 3 — 50줄 미만 코드 변경

자동 스킵. 김대리는 평소대로 코드 변경 결과만 보고하고 다음 진행.

### 예시 4 — 부장님 스킵 명시

부장님: "skip"

김대리는 리뷰 카드 출력 생략하고 다음 단계 진행. 세션 내 다음 산출물 직후에는 다시 카드 출력 의무.
