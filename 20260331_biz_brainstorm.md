## 핵심 결론

### 아이템: 웹 PTT 위젯 SaaS
- 스크립트 한 줄로 어떤 웹 서비스든 PTT 무전 기능 추가
- Intercom이 채팅 위젯 SaaS를 만든 것과 같은 구조
- 전 세계에 웹 브라우저 상용 PTT 없음 (World's First #1)
- PTT 위젯 SaaS도 없음 (World's First #2)

### 사업 전략: 위젯은 미끼, SFU + SDK가 본업
```
위젯(무료) → 입소문 → SFU/SDK 판매(B2B) → 디스패치 SI(최후 카드)
```
- 사용자에게 과금 아님. 위젯 도입한 사업자에게 B2B 계약
- 초기 과금 시스템 불필요. 입금 → app_id 발급

### hub 아키텍처
- 위젯 SaaS 기능(멀티테넌시, 토큰, 채널 관리)은 hub에 모놀리스로
- 병목 생기면 SFU처럼 떼어내는 구조

## 탐색 과정 (기각 포함)

1. **프록터링(시험감독)** → 모니토/나우앤테스트/아르고스 등 이미 레드오션. 기각
2. **건설/경비/물류 직접 서비스** → 파워텔 독주체제. SKT 오키토키도 못 잡음. 기각
3. **디스패치 SI** → SaaS로 못 품. 최후의 카드로 남겨둠
4. **디스패치 분해 (엔진 SaaS + 화면 SI)** → "위젯"으로 발전하는 씨앗
5. **업종별 템플릿 → 직접 서비스** → 업종 찍기 반복 실패. 방향 전환
6. **"아무 화면에 무전 버튼이 자연스럽다"** → 핵심 전환점. PTT = 독립 서비스가 아니라 다른 웹 서비스에 붙는 기능
7. **웹 PTT 위젯 SaaS** → 최종 수렴
8. **웨비나 음성 Q&A** → 수천~수만 명은 SFU 불가(CDN 영역). 기각
9. **AI PTT 음성 인터페이스** → 해자 없음(OpenAI가 버튼 추가하면 끝). 위젯 부가기능으로는 가능
10. **MDM 업체 SFU 납품** → SFU 판매 잠재 고객군으로 기록

## 핵심 인사이트

### PTT vs 통화의 근본적 차이
- 통화: 기본 = 내 소리 나감. 끄려면 행동 필요. 항상 연결 부담 큼
- PTT: 기본 = 조용. 말하려면 행동 필요. 항상 연결 비용 제로
- "Always-on, never awkward"

### 토글 PTT
- 길게 누르는 hold 방식은 긴 발언에 고통
- 토글 한 번 = 발언권 ON, 또 한 번 = 발언권 OFF
- AI 음성 대화에서 turn-taking 문제를 물리적으로 해결

### 키보드 단축키
- 마우스로 버튼 위치까지 이동하는 불편함 제거
- 무전기 PTT 버튼의 본질을 키보드로 이식
- 위젯 조작 = 키보드, 위젯 상태 표시 = 플로팅 인디케이터

### 완성품 → 아이디어 순서
- 보통: 아이디어 → 투자 → 개발 → 실패 (돈 태움)
- OxLens: 완성품이 먼저 → 어디에 팔지 탐색 (돈 안 태움)
- 위젯 안 먹히면 피벗, 엔진은 그대로

### 부장님 배경
- 오키토키 프로(SKT PTT) 시그널링 + 안드로이드 직접 개발
- 국내 PTT 시장(파워텔 독주, ProPTT2/BizPTT/사이버텔브릿지 영세) 실무 경험
- 디스패치 SI 현장 고충 체감 (WinForm 화면 변경 어려움)
- 영업 팀장 객원 확보 완료

## 레드팀 (안 될 수 있는 이유)

1. "아무도 안 한" ≠ "기회". "아무도 안 필요한"일 수 있음 → Product Hunt 검증 필요
2. 빅테크(Twilio/Agora) 진입 가능성 → 선점+전환비용으로 방어
3. 1인 체제 병목 → 성공했을 때의 고민. 매출 나면 사람 씀
4. 웹 브라우저 제약 (모바일 백그라운드 불가) → PC 전용. 모바일은 네이티브 앱
5. Chrome 정책 변화 → Zoom/Meet도 같은 리스크라 가능성 낮음
