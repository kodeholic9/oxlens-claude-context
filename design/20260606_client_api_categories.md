# 클라 SDK 공개 표면 카테고리 대조 (0606d)

> 지침: `claudecode/202606/20260606d_category_crossverify.md` + 보정(C7 신설, PTT 독립 + Scope 편입). **7 카테고리**에 업계 3사 API + 우리 표면을 누락 없이 채운 대조표. 채점/판정 아님 — 빠짐없는 정리.
> **C1~C6 = 업계 객체축**(mediasoup/livekit/jitsi 대응 존재) / **C7 = OxLens 고유 도메인(PTT/무전)** — 3사 칸 전부 `—`, 빈 칸이 곧 차별점.
> 칸은 **소스에서 직접 추출**(README/docs 아님). 빈 칸(있는 쪽만 채워짐)은 `—` 로 보이게 둠. 어디에도 안 들어가는 표면은 맨 밑 "미배치".
> 소스: mediasoup-client `reference/mediasoup-client/src/` · LiveKit `reference/client-sdk-js/src/` · Jitsi `reference/lib-jitsi-meet/` · 우리 `oxlens-home/sdk/`(core/ 레거시 제외).

## 미배치

> 지침 §5 — 7 카테고리 밖이라 부장님 판정이 필요한 것 (임의 재배치 금지, 메모만). 경계 판정 3건은 0606d 보정(C7 신설)으로 전부 닫힘.

**경계 판정 (닫힘 — 0606d 보정):**
- `Scope` / SCOPE(0x1200) 다방청취 → **C7 편입 확정** (PTT/무전 도메인).
- `Device.canProduce()` / `sendRtpCapabilities` (mediasoup) → **C1 유지 확정** (Device 세션 부트스트랩).
- `setAttributes`/`setMetadata` (livekit) → **C5 단일 귀속 확정** (C2 발행메타에서 제거).

**7 카테고리 밖 — 앱 기능(SFU SDK 표면 아님), 대부분 jitsi 단독.** 표는 "무엇/어느 SDK/왜 밖" + **[첨언]** = OxLens(B2B 웹 PTT+Conference, SFU dumb 원칙) 관점 의미. 첨언은 사실 아닌 *포지셔닝 판단* — 부장님 확정 대상.

| 클러스터 | 표면 (SDK) | 왜 6 카테고리 밖 | [첨언] OxLens 의미 |
|---|---|---|---|
| Lobby/입장심사 | `LOBBY_USER_*` / `CONFERENCE_ACCESS_DENIED` (jitsi) | 인가는 SFU 아닌 hub/app 도메인 | hub Extension(JWT+role+moderate)로 대응 가능 — 클라 SDK 표면 아님. 우리 위치 정합. |
| Polls/투표 | `POLL_RECEIVED`/`POLL_ANSWER_RECEIVED` (jitsi) | 순수 앱 데이터 — 미디어 무관 | 우리는 MESSAGE(0x1301)/ANNOTATE(0x1303) 같은 generic relay 위에 앱이 얹는 게 정석. SDK 1급 표면 불요. |
| Transcription/자막 | `TRANSCRIPTION_STATUS_CHANGED`/`getTranscriptionStatus` (jitsi) | STT 외부 서비스 연동 | 미래 토픽이면 telemetry/event 채널 재사용. 현 범위 밖. |
| E2EE verification | `E2EE_VERIFICATION_*`/`startVerification` (jitsi), `EncryptionError`/`isEncrypted` (lk) | insertable streams 종단암호화 | SFU 는 SRTP 종단(복호→라우팅→재암호)이라 **E2EE 와 구조 상충** — PTT gating/rewrite 가 평문 RTP 전제. 도입 시 아키텍처 재검토. 현 기각 영역. |
| AV Moderation | `AV_MODERATION_*` (jitsi) | 발언 자격 관리 | **우리는 이미 대응** — hub Moderate Extension(C5 MODERATE 0x1701, grant/revoke). jitsi 는 이벤트 다수지만 우리 1 op + Extension. 위치 정합. |
| Recording | `RECORDING_STATE_CHANGED` (jitsi), `RecordingStatusChanged` (lk) | 서버측 녹화(Jibri/Egress) | 우리는 서버 oxtapd 자리(**미구현** — system.toml `[recording]` 死섹션, 0606c D-1). 구현 시 클라 표면(상태 이벤트) C6 에 편입 예정. |
| Reactions/Feedback | `sendReaction`/`sendFeedback` (jitsi) | UX/품질설문 앱 기능 | generic 메시지 채널로 충분. SDK 표면 불요. |
| RPC | `performRpc`/`registerRpcMethod` (lk) | 참가자간 요청/응답 추상 | 우리 wire 가 이미 request/ACK(pid) 대칭 — 앱이 MESSAGE 위에 구현 가능. 1급 RPC 표면은 과설계 여지. |
| 파일/바이트 스트림 | `sendFile`/`streamBytes` (lk) | 대용량 신뢰전송 | 우리 DC reliable 채널 = Phase 2. 그때 C5 편입. 현 미배치. |
| Backup codec | `publishAdditionalCodecForTrack`/`backupCodecPolicy` (lk) | 코덱 폴백 정책 | 우리는 서버 코덱 정책 고정(SDP-free)이라 클라 codec 협상 표면 자체가 없음 — 구조적 비대응(빈 칸 정상). |

> 종합 [첨언]: 카테고리 밖 = 거의 다 **(a) 앱-레벨 기능**(Lobby/Polls/Reactions/Feedback — generic relay 위) 또는 **(b) 서버/구조 영역**(Recording=oxtapd / E2EE=SRFU 상충 / Moderation=hub Extension). "SDK 클라 표면이 비어 있음"이 우리 설계(SFU dumb + hub 정책 + 서버 코덱 고정)의 *결과*지 누락이 아님. 미배치 유지가 맞고, 표는 그 비대칭을 가시화하는 게 목적.

**우리만 있고 3사 대응 없음 (빈 칸 정상 — 가시화 목적):**
- **C7 PTT/무전 일체**(Floor/Power/MBCP/Scope/Virtual/Freeze) / duplex 전환(TRACK_STATE_REQ, C2) / 사건보고(CLIENT_EVENT, C6) / 제어 wire 프레이밍(encodeFrame/OutboundQueue, C1·C5). — 미배치 아님(정상 배치), 비대칭 기록용.
